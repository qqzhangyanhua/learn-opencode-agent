# 工具调用

工具调用是 Agent 从“会说”变成“会做”的关键层。面试官通常会在这里判断你是否真的做过 Agent，而不是只会讲概念。下面的结构会同时给你一个能快速作答的版本，以及能展开到执行细节的版本。

## 高频题

### 1. Agent 是怎么决定调用哪个工具的？

- 题目：模型为什么知道该选文件搜索、数据库查询还是网页抓取？
- 面试官想听什么：你是否知道工具选择不是“模型神奇理解”，而是 schema、描述和运行时约束共同作用的结果。
- 短答版：模型主要是根据 prompt、工具描述和参数 schema 来判断该调用什么工具。工程上不能只依赖它“自己懂”，而要靠清晰描述、权限裁剪和反馈机制减少误选。
- 深答版：这题的本质是解释工具选择机制，而不是神化模型。模型在当前上下文里看到可用工具列表、每个工具的用途描述、参数结构和用户目标，然后生成一个最符合语义的调用意图。边界上，工具选择既受模型能力影响，也强依赖你给出的工具定义质量，所以“写好 description 和 schema”本身就是系统设计的一部分。工程取舍在于，工具越多、定义越重叠、参数越模糊，误选概率就越高，因此实际系统里常常会做权限过滤、分场景暴露工具或分层工具目录。常见误区是以为换更强模型就能解决所有误选问题，但如果工具集合本身设计混乱，再强的模型也会被噪声拖偏。
- 追问方向：如果工具很多，为什么模型更容易选错？

### 2. 工具调用闭环的关键步骤是什么？

- 题目：从用户提问到工具结果被模型再次利用，中间发生了什么？
- 面试官想听什么：你是否真正理解工具调用是一个运行时闭环，而不是“一次函数调用”。
- 短答版：典型流程是：用户消息进入上下文，模型输出工具调用意图，运行时解析参数并执行工具，把结果标准化写回消息轨迹，再让模型基于新结果继续决策。关键不只是调用成功，而是结果能被下一轮推理消费。
- 深答版：这题本质上是在考你有没有把 Agent 当成状态机来看。真正的工具调用闭环一般包含五步：读取当前上下文、让模型生成工具意图、在运行时校验并执行、把结果结构化写回、让模型继续下一轮决策。边界上，工具不是直接替模型思考，而是给模型提供新的外部观察，所以结果回填格式非常关键。工程取舍在于，如果结果回填太随意，模型下一轮就容易丢信息；如果回填太长，又会抬高上下文成本。常见误区是把焦点全放在“怎么调用工具”，但系统真正稳定与否，往往取决于“结果怎么回填”和“失败怎么反馈”。
- 追问方向：如果工具结果格式不稳定，会导致什么问题？

### 3. 为什么权限系统不能只靠 prompt 约束？

- 题目：如果 prompt 里写“不要删文件”，是不是就够了？
- 面试官想听什么：你能不能把“行为建议”和“硬性约束”分开，不会把安全边界寄托在提示词上。
- 短答版：不够。prompt 只能影响模型倾向，不能形成硬约束。真正的安全边界必须落在运行时权限层，比如白名单、参数校验、人工确认和沙箱执行。
- 深答版：这题的核心判断是，prompt 属于软约束，权限系统属于硬约束。软约束能提高模型遵守规则的概率，但不能在异常输入、提示词污染或模型误判时提供真正的阻断能力。边界上，凡是涉及删除、执行、联网、写数据库这类不可逆或高风险操作，都应该由运行时在工具层做最终裁决。工程取舍在于，硬约束会牺牲一部分灵活性，但换来可审计性和可控性，这在生产场景几乎是必需的。常见误区是把“系统提示词写得很严格”当成安全设计，但一旦用户输入污染上下文，或者模型本身误判，这种防线很脆。
- 追问方向：哪些操作最值得加人工确认？

### 4. 工具失败时应该怎么处理？

- 题目：如果 API 超时、命令执行失败、参数不合法，Agent 应该怎么办？
- 面试官想听什么：你是否会先分类失败，再决定恢复策略，而不是所有失败都一把梭哈重试。
- 短答版：先区分失败类型。可重试错误按策略重试，参数错误应尽快反馈给模型或用户修正，权限错误要直接暴露边界。关键是把失败转成结构化反馈，而不是只吐一段日志。
- 深答版：这题本质是在看你有没有恢复策略。工具失败至少可以分成临时性失败、输入错误、权限拒绝和系统异常四类，不同类型的处理方式完全不同。边界上，临时网络错误可以有限重试，但参数错误和权限错误通常不该重试，因为那只会浪费 token 和时间。工程取舍在于，你既要给模型足够的信息帮助它改正下一步，又不能把底层异常原样塞回去污染上下文，所以结构化错误对象很重要。常见误区是所有失败都自动重试三次，但如果错误根因是参数错或权限不够，重试只会放大成本和混乱。
- 追问方向：什么情况下重试反而会放大问题？

### 5. 工具系统怎么避免越做越乱？

- 题目：如果后面要不断加工具，怎么保证系统还可维护？
- 面试官想听什么：你有没有“可扩展工具平台”视角，而不是只会在 demo 里硬塞几个函数。
- 短答版：重点是统一注册、统一 schema、统一执行入口和统一结果回填。不要把工具能力散落在 if/else 里，先稳定数据结构，后面新增工具才不会变成复制粘贴工程。
- 深答版：这题的核心是平台化思维。工具系统一开始可能只有几个工具，但只要它真能用，后面一定会不断加能力，这时最重要的是统一接口而不是单个工具写得多巧。边界上，工具的职责应该是“声明能力 + 校验输入 + 返回结构化结果”，而不是每个工具自带一套特殊调度逻辑。工程取舍在于，统一注册和 schema 会让前期多写一点约束代码，但能极大降低后期维护成本、权限治理成本和调试难度。常见误区是为了快，在主流程里到处加特判，短期看省事，长期会把工具系统做成无法治理的补丁集合。
- 追问方向：为什么“工具描述设计”本身就是系统质量的一部分？

## 扩展题

### 6. 为什么工具 schema 质量会直接影响 Agent 效果？

- 题目：同一个工具能力，如果 schema 写得很粗糙，会具体坏在哪？
- 面试官想听什么：你是否知道 schema 不只是参数校验层，还会反向影响模型理解、调用稳定性和错误恢复。
- 短答版：schema 质量差，模型就更容易误解参数语义、漏字段、乱填默认值，最后表现成误调用和高重试率。它不是文档细节，而是工具可用性的核心接口。
- 深答版：这题的本质是把 schema 看成“给运行时校验的结构”，还是“给模型理解的契约”。如果字段命名含糊、枚举范围不清、必填和选填边界不明，模型在决策时就会出现歧义，后面再靠补丁校验只是在事后救火。边界上，好的 schema 应该同时服务两件事：让模型更容易一次选对参数，也让运行时更容易识别错误并给出可恢复反馈。工程取舍在于，前期把 schema 写细会增加一点定义成本，但能显著降低误调用、无效重试和提示词补丁数量。常见误区是把 schema 当成“能过 JSON 校验就行”，但对 Agent 来说，语义表达不清本身就是系统故障源。
- 追问方向：字段命名、枚举设计和必填项约束，哪个最容易影响模型误判？

### 7. 为什么不能把所有工具都直接暴露给模型？

- 题目：工具越多是不是能力越强，为什么很多系统反而做按场景暴露？
- 面试官想听什么：你能不能从治理视角回答工具暴露问题，而不是只从“功能更多”出发。
- 短答版：不是越多越强。工具暴露面越大，模型选择噪声越大，越权风险和调试成本也越高，所以生产里通常按角色、任务和风险等级裁剪工具集。
- 深答版：这题的核心判断是，工具暴露本身就是治理动作，不是简单功能开关。模型看到的候选工具越多，语义重叠越严重，选择错误率通常越高；同时一旦高风险工具被无差别暴露，安全和审计压力也会立刻上来。边界上，读操作、写操作、执行类工具不该放在同一层级随意开放，而应该根据任务阶段、用户权限和环境风险做动态裁剪。工程取舍在于，少暴露工具会牺牲一点“全能感”，但换来更稳定的路由、更清晰的权限边界和更低的事故面。常见误区是把工具目录做成应用商店，觉得模型自己会挑，但真实系统里这通常只是把治理问题推给模型。
- 追问方向：你会按哪些维度裁剪工具暴露范围？

### 8. 什么时候该并行调工具，什么时候必须串行？

- 题目：如果一个任务里要查多个外部信息，怎么判断是并行还是串行更合适？
- 面试官想听什么：你是否理解并行不是默认更优，而是取决于依赖关系、成本和失败语义。
- 短答版：彼此独立、结果只做汇总的查询适合并行；后一步依赖前一步结果、或者涉及副作用和确认链路时，就必须串行。核心不是追求快，而是先看依赖图。
- 深答版：这题本质上是在考调度判断。并行调用的价值在于压缩总等待时间，但前提是各工具之间没有数据依赖，也不会因为同时执行放大副作用或资源争抢。边界上，只读查询、多源比对、互不影响的检索适合并行；而任何需要先拿到上一步结果才能决定参数、权限或动作的场景，都应该串行。工程取舍在于，并行能换延迟，但会抬高并发控制、超时处理和错误聚合复杂度；串行更稳，但可能拖慢整体响应。常见误区是一味追求“能并就并”，结果把原本清晰的决策链做成难回滚、难解释的并发补丁。
- 追问方向：并行调用里如果只失败一个工具，结果该怎么汇总才不误导模型？

### 9. 为什么外部写操作工具要强调幂等性？

- 题目：创建工单、发消息、扣库存这类工具，为什么要单独谈幂等而不是只谈重试？
- 面试官想听什么：你是否意识到 Agent 不只是读世界，还会改世界，所以副作用控制比“调用成功”更关键。
- 短答版：因为写操作一旦重试，就可能重复扣费、重复下单、重复发通知。幂等不是优化项，而是防止 Agent 把临时失败放大成业务事故的底线。
- 深答版：这题的核心是把“模型可重试”与“业务可重试”分开。模型层看到超时，可能会认为上次没成功，但真实世界里那次请求也许已经生效了，如果没有幂等键或去重机制，再来一次就是重复副作用。边界上，只读工具最多浪费成本，写操作工具则会改状态、触发通知甚至产生资金影响，所以必须把请求标识、去重策略和结果查询机制设计进去。工程取舍在于，幂等设计会增加一点接口复杂度，但它换来的是可恢复性和可审计性，尤其适合 Agent 这种天然会重试、会补调用的系统。常见误区是把幂等交给模型“自己记住不要重复调用”，这在工程上根本站不住。
- 追问方向：如果外部系统本身不支持幂等，你会怎么补救？

### 10. 工具结果回写为什么会污染后续推理？

- 题目：既然工具结果要给模型继续用，为什么不能把原始输出完整塞回上下文？
- 面试官想听什么：你是否理解结果回写不是日志转存，而是给下一轮推理准备可消费信息。
- 短答版：原始输出往往太长、太脏、太像系统内部噪声，直接回写会稀释真正关键信息，还可能把错误字段和无关细节带进后续决策。回写要做摘要、结构化和风险过滤。
- 深答版：这题本质上是在谈“写回什么”比“拿到什么”更重要。工具原始结果通常包含调试字段、重复片段、低价值元数据，甚至混着异常堆栈和供应商噪声，这些内容一旦无差别进入上下文，就会抬高 token 成本、干扰注意力分配，还可能诱导模型基于脏数据继续推理。边界上，结果回写应该保留对下一步决策真正必要的事实、状态和失败信号，而不是把底层返回体当作 transcript 全量存档。工程取舍在于，做结构化提炼会多一道处理，但能显著降低上下文污染和后续误判。常见误区是为了“信息别丢”把一切都塞回去，最后系统不是更聪明，而是更嘈杂。
- 追问方向：哪些工具结果适合只回写摘要，哪些必须保留原始字段？

### 11. 工具超时预算该怎么分配？

- 题目：Agent 一条链路里可能调很多工具，超时是给每个工具统一设一个值，还是按阶段分配？
- 面试官想听什么：你是否有整体预算意识，而不是把超时看成单个接口参数。
- 短答版：我一般按整条任务预算倒推，而不是每个工具拍脑袋设统一超时。不同工具的价值、稳定性和可替代性不同，预算也应该不同。
- 深答版：这题的核心判断是，超时预算属于链路调度问题，不只是网络参数。用户能接受的总等待时间通常是固定的，所以你需要先定义总预算，再分给检索、执行、重试和模型推理这些阶段，而不是让每个工具各自慢到底。边界上，关键路径工具可以拿更多预算，不关键或可替代工具应该更早失败并触发降级；并行调用还要考虑聚合等待策略，避免一个慢工具拖住整轮决策。工程取舍在于，预算收得太紧会增加误失败，放得太松又会拉高整体延迟和资源占用，所以要结合历史耗时分布、业务 SLA 和替代方案来定。常见误区是给所有工具统一 30 秒，看起来公平，实际上既不高效，也不符合任务优先级。
- 追问方向：超时之后应该重试、降级还是直接暴露失败，怎么判断？

### 12. 核心工具不可用时，Agent 应该怎么降级？

- 题目：如果搜索、数据库或执行工具临时不可用，系统该直接报错，还是继续给答案？
- 面试官想听什么：你能不能把“兜底回答”和“假装完成”分开，知道什么场景能降级，什么场景必须停。
- 短答版：要按任务风险分级。低风险场景可以降级到保守回答或请求用户补充，高风险和需要外部确认的场景必须明确中止，不能编一个看起来完整的结果。
- 深答版：这题本质上是在谈降级策略的边界。降级不是“工具坏了也照常回答”，而是当外部能力缺失时，系统如何退回到一个仍然诚实、可控、风险可接受的模式。边界上，信息查询类任务可以退到基于已有上下文的保守回答、局部缓存或人工接管；但涉及资金、写操作、权限确认或高真实性要求的任务，如果关键工具不可用，就应该显式失败。工程取舍在于，降级能提高可用性，但如果没有风险分层，就很容易把“服务不中断”做成“错误更隐蔽”。常见误区是把降级理解成不惜一切代价继续输出，这对 Agent 系统尤其危险，因为用户很难第一时间看出它是在猜。
- 追问方向：你会怎么设计“允许降级”和“禁止降级”的判定规则？
