# 工程化

这一类题最能区分“做过 Demo”和“做过系统”。面试官会借它判断你是否有把 Agent 放进真实业务环境的能力。短答版帮你先说结论，深答版则展开评估、安全、监控、成本和恢复这些真正落地的问题。

## 高频题

### 1. Agent 怎么做评估？

- 题目：传统服务能看接口正确率，Agent 这么开放，评估该怎么做？
- 面试官想听什么：你是否知道 Agent 评估不能只盯最终答案，而要分层看过程和结果。
- 短答版：我会做分层评估。底层看工具正确性和权限边界，中层看任务链路是否按预期推进，高层看最终任务完成质量；不能只看一句最终回答像不像对。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的本质是建立 Agent 的评估分层。Agent 是开放系统，同一个任务可能走不同路径，所以如果只看最后结果，你很难知道问题出在工具层、规划层还是模型决策层。边界上，底层更关注确定性行为是否正确，中层关注链路是否稳定推进，高层才看最终质量和用户价值。工程取舍在于，分层评估需要更多测试集、标注和日志，但它能显著提升迭代效率，因为你知道该修哪一层。常见误区是把 Agent 评估做成“问几个问题看答案像不像”，这种方式既不稳定，也不支持系统性优化。
  </details>
- 追问方向：离线评估和线上监控分别适合抓什么问题？

### 2. Agent 上线后最该监控什么？

- 题目：如果只能先打几类核心监控，你会优先监控哪些指标？
- 面试官想听什么：你是否知道生产环境里哪些信号最能反映 Agent 是否真的可用。
- 短答版：我会先盯任务完成率、工具失败率、人工接管率、延迟、token 成本和高风险操作触发情况。这些指标能同时反映质量、效率和风险。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的核心判断是，Agent 监控必须兼顾结果、过程和风险。只看平均响应时间远远不够，因为一个 Agent 可能响应很快但任务经常失败，也可能成功率高但成本失控。边界上，任务完成率和人工接管率更反映业务价值，工具失败率和高风险操作更反映系统健康，延迟和 token 成本则决定可运营性。工程取舍在于，第一阶段先打少量高价值指标，比一口气采集海量日志更实用，因为你要优先建立告警和定位能力。常见误区是照搬传统 Web 服务监控指标，但 Agent 系统最大的风险常常发生在“链路推进质量”而不是单个接口延迟。
  </details>
- 追问方向：为什么只看平均响应时间对 Agent 不够？

### 3. Agent 的安全边界怎么做？

- 题目：如果 Agent 能读文件、跑命令、访问网络，怎么避免越权？
- 面试官想听什么：你是否把安全理解成系统级防线，而不是“写一个很严格的系统提示词”。
- 短答版：核心是把边界放在运行时，而不是 prompt。常见做法包括工具白名单、参数校验、人工确认、沙箱执行、审计日志和最小权限原则，安全要做成分层防线。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的本质是安全边界必须系统化。只要 Agent 能访问文件、命令和网络，它就已经具备高风险能力，所以你必须假设模型会误判、输入会污染、外部环境会返回恶意内容。边界上，安全不是单一功能，而是多层叠加：工具暴露范围、参数约束、执行沙箱、人工确认、操作审计和异常回滚一起构成防线。工程取舍在于，这些护栏会减少一部分“全自动”的流畅度，但换来的是可控性和上线资格。常见误区是把 Prompt Injection 理解成 prompt 写得不够好，但它本质上是输入污染问题，必须靠系统边界而不是文案边界解决。
  </details>
- 追问方向：Prompt Injection 为什么本质上是“输入污染问题”，不只是提示词问题？

### 4. Agent 的成本怎么控？

- 题目：如果系统功能可以跑，但 token 成本太高，你优先从哪下手？
- 面试官想听什么：你是否知道成本控制要从结构入手，而不是一句“换便宜模型”。
- 短答版：我会先找大头，通常是长上下文、无效重试、低价值大模型调用和召回冗余。然后再用模型路由、上下文裁剪、缓存和更稳的失败处理去收缩成本。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的核心判断是，成本问题通常是系统设计问题，不只是模型价格问题。很多 Agent 成本高，不是因为模型太贵，而是因为上下文太长、计划过度、重试太多、每一步都用最强模型，或者召回和回填信息严重冗余。边界上，降模型是一个手段，但如果不先定位成本大头，很容易牺牲质量却省不了多少钱。工程取舍在于，你需要把昂贵调用留给真正高价值节点，把便宜模型、缓存、裁剪和批处理用在能承接的地方。常见误区是一看到成本高就统一降配，但如果主问题是上下文浪费和失败重跑，单纯降模型往往会让质量更差、总成本反而更高。
  </details>
- 追问方向：为什么“多模型路由”通常比“一把梭哈最强模型”更适合生产？

### 5. Agent 出错后怎么恢复？

- 题目：如果任务中途失败，怎么避免整条链路从头再来？
- 面试官想听什么：你是否意识到“恢复能力”是生产系统和 demo 的分水岭。
- 短答版：工程上要有可恢复状态，而不是只有最终自然语言答案。常见做法包括持久化中间状态、保存工具结果、明确阶段边界和支持人工接管。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题本质是在谈系统是否具备连续运营能力。Demo 可以失败后从头再跑，但生产系统如果每次异常都重来，不仅成本高，很多外部操作还可能不可逆。边界上，恢复能力依赖于状态持久化、阶段边界、工具结果记录和人工介入点，这些都属于运行时设计，不是最后补个日志就能解决。工程取舍在于，做恢复会增加状态管理复杂度，但它能大幅降低失败代价、排障难度和人工成本。常见误区是只保存最终回答，不保存中间轨迹，结果一旦任务中途出错，系统既无法断点续跑，也很难解释刚才到底发生了什么。
  </details>
- 追问方向：如果没有状态持久化，哪些故障会特别难排查？

## 扩展题

### 6. Agent 发布前要设哪些门禁？

- 题目：如果一个 Agent 功能准备上线，你会设置哪些 release guardrails，避免“能跑但不敢发”？
- 面试官想听什么：你是否知道上线不是点按钮，而是先定义哪些风险绝对不能带进生产。
- 短答版：我会把门禁分成质量、安全、成本和回退四层。至少要有关键用例回放通过、高风险工具受控、成本没有明显漂移、失败时能快速止损，不满足就不放量。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题核心不是“上线前多测一点”，而是把上线标准做成明确门槛。Agent 的问题是看起来能回答，但一进真实流量就可能在权限、成本、链路稳定性上失控，所以门禁必须覆盖结果质量、危险操作、预算上限和回退能力。边界上，门禁不是越多越好，太重会拖慢迭代，太轻又等于裸奔；通常做法是只卡最容易引发生产事故的点，比如关键任务集回放、危险操作审批、异常率上限、版本可切回。常见误区是把门禁理解成人工拍板，真正可持续的是把它变成发布流程里的硬条件。
  </details>
- 追问方向：哪些指标适合做硬门禁，哪些更适合上线后持续观察？

### 7. 回放评估为什么重要？

- 题目：为什么很多团队会做 replay-based evaluation，而不是只靠人工 spot check？
- 面试官想听什么：你是否理解回放评估的价值在于复现实流量、定位回归，而不只是多跑一轮测试。
- 短答版：因为 Agent 的问题很多出在线上真实输入分布，人工抽查覆盖太差。回放评估能把历史任务按新版本重跑，对比成功率、轨迹、成本和风险信号，抓回归更稳。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题本质是在谈评估数据是否贴近生产。静态题库适合做底线检查，但 Agent 很多退化来自真实用户输入、上下文噪声和工具返回的不稳定性，这些人工 spot check 很难覆盖。边界上，回放不是为了百分百模拟线上，而是用可复现的历史样本比较新旧版本，重点看成功率变化、关键步骤偏移、成本上涨和高风险操作触发是否异常。工程取舍在于，回放体系要保存输入、上下文快照和工具响应，建设成本不低，但它能大幅降低“改了一点小东西却把旧场景打坏”的风险。常见误区是只比最终回答文本，忽略中间轨迹，结果很多决策退化根本看不出来。
  </details>
- 追问方向：如果工具返回具有时效性，回放评估怎么避免失真？

### 8. Agent 告警应该怎么设计？

- 题目：Agent 系统的告警要怎么做，才能既能早发现问题，又不把值班人淹没？
- 面试官想听什么：你是否知道告警设计的重点是可行动，而不是把所有异常都推给人。
- 短答版：我会按可行动性设计告警。优先报任务完成率突降、高风险操作异常、工具连续失败、人工接管飙升这类必须处理的信号，低价值抖动进看板不直接打人。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题核心是告警不是日志搬运。Agent 的异常很多，如果什么都告，最后没人信告警；如果只告基础设施，又抓不到链路质量退化。边界上，好的告警要同时满足三点：和用户影响强相关、能定位到责任层、收到后有明确动作。工程取舍在于，通常把任务成功率、危险动作触发、关键工具错误、预算异常做成高优先级，把长尾波动、普通重试和非关键质量波动先沉到报表。常见误区是直接照搬接口 5xx 告警，但 Agent 更常见的是“接口都正常，任务却持续做错”。
  </details>
- 追问方向：任务完成率和模型输出质量难以实时判断时，告警阈值怎么设？

### 9. Agent 的审计能力要做到什么程度？

- 题目：如果用户投诉 Agent 做了不该做的事，你认为系统至少要具备哪些 auditability 能力？
- 面试官想听什么：你是否把审计理解成可追责、可解释、可复盘，而不是简单留一份聊天记录。
- 短答版：至少要能回答四件事：谁触发的、当时看到了什么、调用了哪些工具、为什么落成了这个结果。也就是输入、关键决策、执行动作和版本信息都要能追到。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题本质是在谈系统出了事以后能不能说清楚。Agent 一旦接入外部工具，问题就不只是“答错一句话”，还可能涉及误操作、越权和责任归属，所以审计必须覆盖请求身份、上下文快照、模型版本、工具调用参数、人工确认点和最终结果。边界上，审计不等于无限制记录一切，你还要处理隐私、存储成本和敏感信息脱敏，所以关键是保留能复盘决策链的最小必要证据。工程取舍在于，审计越完整，排障和合规越稳，但日志治理成本会上升；真正危险的是只有最终输出，没有过程证据。常见误区是出了问题才想补日志，那时通常已经来不及。
  </details>
- 追问方向：审计日志和普通观测日志的边界应该怎么划？

### 10. 回滚设计为什么不能只靠重新部署旧版本？

- 题目：如果新版本 Agent 上线后效果变差，为什么“把代码回滚”往往还不够？
- 面试官想听什么：你是否意识到 Agent 的状态、配置、提示词、工具协议都可能让回滚变复杂。
- 短答版：因为 Agent 行为不只由代码决定，还受 prompt、模型版本、工具 schema、策略配置和中间状态影响。真正的 rollback design 要能一起切回这些依赖，并处理已经进行中的任务。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题核心是 Agent 的运行面比传统服务更宽。很多事故不是代码 bug，而是 prompt 更新、模型切换、工具字段变化、阈值改动或者记忆污染导致的，所以只回滚应用镜像，行为未必能回到旧状态。边界上，回滚至少要覆盖版本配置、依赖协议和在途任务策略，比如老任务继续按旧规则跑，还是统一终止重试。工程取舍在于，回滚面做得越完整，系统越稳，但配置管理和兼容成本会增加；如果完全不设计，线上出问题时就只能一边救火一边猜原因。常见误区是默认“重新 deploy 就算回滚”，这在 Agent 场景里经常是假恢复。
  </details>
- 追问方向：如果新旧版本的工具参数结构不兼容，回滚时最容易踩什么坑？

### 11. Agent 为什么适合做版本化放量？

- 题目：Agent 新能力上线时，你会怎么做 versioned rollout，而不是一次性全量替换？
- 面试官想听什么：你是否知道 Agent 质量波动大，发布策略本身就是风险控制手段。
- 短答版：我会做版本化放量，按用户组、场景或流量比例逐步开。核心不是只看整体成功率，而是比较新旧版本在关键场景、成本和风险触发上的差异，随时能停。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题本质是在谈把不确定性关进小范围。Agent 很容易在某些场景变好、另一些场景变差，所以一次性全量替换会把问题直接扩散到全站。边界上，版本化放量通常会同时保留旧版本作为对照，按用户分层、任务类型或流量比例逐步扩大，观察真实差异后再决定是否继续。工程取舍在于，多版本并行会增加路由和观测复杂度，但它换来的是更强的比较能力和更低的事故半径。常见误区是只看全局平均指标，如果新版本把一个高价值场景打坏，平均值可能还看不出来。
  </details>
- 追问方向：按用户分桶、按场景分桶、按流量比例分桶，各自适合什么情况？

### 12. 人工接管和恢复演练该怎么准备？

- 题目：怎么判断一个 Agent 系统真的具备 human handoff 和 recovery readiness，而不是文档里写了“必要时人工介入”？
- 面试官想听什么：你是否理解人工接管不是兜底口号，而是要提前设计入口、上下文和操作流程。
- 短答版：关键看三件事：什么时候交给人、交给人时能看到什么、接手后怎么继续处理。没有明确触发条件、上下文摘要和演练过的 SOP，人工接管通常只是写在方案里。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题核心是在谈恢复准备度，而不是恢复意愿。很多团队口头上都说“失败就转人工”，但真正出事时，人拿不到上下文、不知道前面做了什么、也没有安全的补救动作，最后只是把故障换个方式放大。边界上，人工接管至少要包含触发阈值、任务快照、关键决策摘要、可继续操作的界面或脚本，以及事后重新回到自动流的条件。工程取舍在于，这套机制会增加流程设计和培训成本，但它直接决定事故时能否把损失压住。常见误区是把人工接管设计成一个泛化按钮，没有上下文，没有分类动作，也没有恢复演练。
  </details>
- 追问方向：哪些故障最适合直接转人工，哪些更适合自动重试或自动回滚？
