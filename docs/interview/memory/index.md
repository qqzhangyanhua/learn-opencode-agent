# 记忆

很多候选人会把“聊天历史”“RAG”“长期记忆”混成一个概念，这一页就是专门用来拆边界的。你可以先背短答版，再用深答版补足分层、污染和召回策略这些面试里最容易被追问的点。

## 高频题

### 1. Agent 的记忆和上下文窗口有什么区别？

- 题目：既然模型调用时都会带上下文，为什么还需要单独谈记忆？
- 面试官想听什么：你能不能把模型接口限制和上层系统设计拆开讲，而不是把“历史很多”直接等于“记忆很好”。
- 短答版：上下文窗口是单次模型调用能接收多少输入的限制，记忆则是跨调用保存、筛选和召回信息的策略。前者是模型能力边界，后者是系统设计问题。
- 深答版：这题的本质是区分“容量”与“机制”。上下文窗口只是模型一次能看到多少内容，它不负责判断哪些历史值得保留；记忆系统则负责写入、筛选、召回和注入，是一套长期状态管理策略。边界上，系统可以有远超上下文窗口的历史和事实，但真正送进模型的只是其中的一小部分，因此“有历史”不等于“有记忆能力”。工程取舍在于，如果你把所有历史都硬塞进上下文，成本高、噪声多、相关性差；如果完全不做记忆，又会丢失长期连续性。常见误区是把上下文越拼越长当成记忆增强，但这通常只是低效地消耗 token。
- 追问方向：如果把所有历史都直接拼进去，会遇到哪些问题？

### 2. 短期记忆和长期记忆该怎么分？

- 题目：你怎么区分 session 内记忆和跨 session 记忆？
- 面试官想听什么：你是否知道不同信息生命周期不同，不能把所有信息放进一个统一存储桶里。
- 短答版：短期记忆主要服务当前任务，强调局部连续性；长期记忆用于跨会话保留稳定事实、偏好和长期状态。分层的意义是把临时上下文和长期事实拆开治理。
- 深答版：这题的核心判断是，信息不是都值得保存同样久。短期记忆更像当前 session 的工作台，保存的是为了让任务不断档；长期记忆更像用户画像或系统账本，保存的是跨会话仍然有价值的信息。边界上，任务中间状态、暂时结论和局部猜测通常更适合短期层；用户偏好、组织知识、稳定配置更适合长期层。工程取舍在于，分层会增加设计复杂度，但能降低污染和误召回，因为不同层的数据质量要求和过期策略完全不同。常见误区是把所有历史都持久化成长期记忆，结果系统越跑越脏。
- 追问方向：哪些信息适合长期保存，哪些应该过期？

### 3. RAG 和记忆系统是不是一回事？

- 题目：如果系统已经有向量检索了，还需要记忆层吗？
- 面试官想听什么：你能不能把“知识检索”和“状态连续性”这两条线分开，不会把所有检索都叫记忆。
- 短答版：RAG 更偏知识召回，目标是从外部语料里找材料；记忆系统更偏状态连续性，目标是保留关于用户、任务和历史过程的信息。它们可以共用检索技术，但解决的问题不同。
- 深答版：这题本质上是在分清“知识”和“状态”。RAG 的重点是把外部文档、知识库、代码库等材料在需要时召回给模型，它关注的是知识可达性；记忆系统关注的是这个用户、这个会话、这个任务在持续交互中的连续性。边界上，企业知识库文档更像 RAG，用户偏好和任务阶段更像记忆，但它们底层都可能用到向量检索、关键词检索或数据库查询。工程取舍在于，如果你把所有东西都放进同一个检索池，系统会失去类型边界，很难做精确召回和生命周期管理。常见误区是觉得“已经有向量库了，所以记忆问题解决了”，但向量库只是存取手段，不是记忆策略本身。
- 追问方向：用户偏好、任务状态、企业知识库这三类信息分别该放哪？

### 4. 记忆污染是怎么发生的？

- 题目：为什么 Agent 的长期记忆越多，不一定越聪明？
- 面试官想听什么：你是否理解记忆系统最大的难点在质量治理，而不是存储容量。
- 短答版：如果记忆写入没有筛选，错误信息、过时信息和局部上下文就会被反复召回，污染后续决策。记忆系统真正难的不是能不能存，而是该存什么、删什么、降权什么。
- 深答版：这题的核心判断是，记忆污染通常来自“低质量写入 + 无差别召回”。一旦系统把临时猜测、错误结论、过时事实甚至用户随口一句话都沉淀为长期记忆，后续模型就会把这些噪声当成高价值事实来使用。边界上，污染不只是错误内容本身，还包括信息版本冲突、语义不完整和来源不可信。工程取舍在于，要增加写入审核、置信度、版本信息和过期策略，这会多一点治理成本，但能换来长期稳定性。常见误区是把记忆看成“越多越好”的缓存层，结果系统越用越不可信。
- 追问方向：你会用哪些机制降低污染风险？

### 5. 记忆召回怎么做得更稳？

- 题目：如果召回太多无关记忆，会影响回答质量，你怎么处理？
- 面试官想听什么：你有没有一套实际的治理抓手，而不是只会说“优化召回”。
- 短答版：我一般会从写入筛选、召回排序、类型分层和时间衰减四个方向下手。先减少垃圾写入，再给召回结果加来源和置信度，最后控制注入量，避免稀释模型注意力。
- 深答版：这题本质上在问“记忆系统怎么从能用变成好用”。更稳的召回通常不是靠单点魔法，而是靠整条链路治理：写入时筛掉低价值内容，召回时按类型和相关性排序，注入时控制数量和格式。边界上，你要同时考虑相关性和可信度，因为高度相关但来源不可靠的信息同样危险。工程取舍在于，精细排序、分层检索和去重会增加一些系统复杂度，但它比简单地“多召回一些让模型自己选”更省成本也更稳。常见误区是盲目追求 recall，把大量边缘相关内容一起塞给模型，最后反而让真正有价值的信息被淹没。
- 追问方向：为什么“少量高相关召回”通常比“大量历史拼接”更有效？

## 扩展题

### 6. 记忆写入策略该怎么定？

- 题目：用户说过的话很多，你怎么决定哪些要写成长期记忆？
- 面试官想听什么：你是不是知道长期记忆的难点首先在写入门槛，而不是先把一切存下来再说。
- 短答版：我不会默认全量写入，而是按稳定性、复用价值和风险来判断。只有跨会话还成立、后续大概率会用到、且误写成本可控的信息，才值得入长期记忆。
- 深答版：这题核心在写入策略，而不是存储技术。一个实用做法是先做类型判断，再做价值判断：像稳定偏好、长期目标、账户约束这类高复用信息适合写入；临时情绪、一次性任务细节、模型猜测则不该直接持久化。边界上，写入策略最好是“默认保守”，因为错写的代价通常高于漏写，漏写最多影响连续性，错写会持续污染后续回答。工程取舍在于，严格写入会少记一些内容，但系统更稳；宽松写入看似更聪明，长期却更容易积累噪声。
- 追问方向：如果用户一句话里既有稳定偏好也有临时诉求，你会怎么拆？

### 7. 用户要求忘记某些信息时，系统该怎么处理？

- 题目：如果用户说“把我之前的偏好删掉”，你觉得记忆系统需要做到什么程度？
- 面试官想听什么：你能不能把删除请求从“前端提示一下”上升到真正的数据治理问题。
- 短答版：至少要做到可定位、可删除、可验证，并且区分在线服务不再使用和底层存储真正清理。只是在 prompt 里说“别再提了”不算完成删除。
- 深答版：这题重点是“忘记”不能只做表面屏蔽。一个合格的方案要先知道这条用户记忆在哪里，被哪些索引、缓存或摘要引用过，再决定是硬删除、软删除还是墓碑标记。边界上，用户侧删除至少要保证后续不再召回，并清掉派生出来的用户记忆表示；但审计或合规日志通常属于另一类受控数据，是否保留要看独立的留存政策和访问边界，不能和用户记忆走同一条删除规则。工程取舍在于，彻底清理用户记忆链路实现更复杂，但能降低信任和误召回风险；只做逻辑忽略实现简单，却很容易留下隐患。面试里要明确说，删除策略至少要覆盖原始记录、派生表示和缓存路径，同时区分合规留痕数据。
- 追问方向：哪些派生表示必须同步清理，哪些留痕数据应该走单独的保留策略？

### 8. 记忆的 ownership 边界怎么划？

- 题目：用户信息、任务状态、团队知识都能叫“记忆”，到底谁有权创建、修改和删除？
- 面试官想听什么：你是否真的理解 ownership，不只是把数据分层，而是能讲清写入 authority 和治理责任。
- 短答版：我会先定义 owner，再定义谁能对该域做增删改。用户私有记忆应主要由用户或受托代理更新，任务状态由任务执行链路维护，共享知识由组织侧流程治理，不能谁方便谁就写。
- 深答版：这题核心不是“放哪”，而是“谁说了算”。同样叫记忆，用户偏好、会话摘要、任务状态、团队规则的 owner 完全不同，所以写入 authority 也不能一样。边界上，用户私有记忆不能被另一个用户或普通共享流程随意改写；共享规则也不能被单次会话里的模型总结直接覆盖。工程取舍在于，给每个域加 owner、写权限和删除责任会增加系统复杂度，但能避免最危险的问题: 错域写入和越权更新。一句话讲就是，记忆系统不仅要会召回，还要能回答“这条记忆是谁的，谁能改，谁来删”。
- 追问方向：如果模型自动总结想写入长期记忆，哪些域必须额外审批或确认？

### 9. 过时记忆和版本冲突怎么处理？

- 题目：如果系统记住“用户喜欢 A”，后来用户又明确改成“现在改用 B”，你怎么避免旧记忆继续影响回答？
- 面试官想听什么：你能不能把记忆当成会演化的数据，而不是写进去就永远有效的事实。
- 短答版：我会把记忆设计成可版本化、可失效的记录，而不是只存一段文本。新信息到来时，要能覆盖、降权或标记旧版本失效。
- 深答版：这题考的是你是否理解“记忆不是静态事实仓库”。稳定偏好、配置和规则都可能变化，所以存储时最好带上时间、来源、版本状态甚至 superseded 关系。边界上，如果只把记忆当一段自然语言塞进向量库，就很难处理“旧的还像真话，但已经不该用了”的问题。工程取舍在于，做版本管理会多一点元数据和更新逻辑，但能明显减少冲突召回；不做版本只靠相似度检索，系统很容易把新旧偏好一起带出来，让模型自己猜。面试里可以直接说，冲突记忆默认不该并存为同级事实。
- 追问方向：你会优先用时间戳、显式版本号，还是状态位来解决冲突？

### 10. 记忆要不要带置信度？

- 题目：如果一条记忆来自模型总结，而不是用户明确表达，你会怎么使用它？
- 面试官想听什么：你能不能区分“事实”“推断”“猜测”，并把这种差异体现在系统设计里。
- 短答版：我不会只看一个 confidence 字段，而是先看来源类型、证据强弱、有没有用户确认。置信度可以有，但只是辅助信号，不能假装成精确真值。
- 深答版：这题核心不是“要不要打分”，而是要不要把不同证据强度混成同一种事实。用户明确陈述、系统结构化配置、模型总结、第三方同步数据，这几类来源的可靠性本来就不同；再往下还要看有没有原始证据、有没有被用户确认。边界上，如果系统只依赖一个看起来很精确的 confidence 数值，反而容易制造假确定性，因为这个分数本身也常常是估出来的。工程取舍在于，增加来源类型、确认状态和证据链会让实现更复杂，但比单纯迷信分数更稳。面试里可以直接说，低证据记忆可以保留成候选线索，但不该直接当高优先级事实写回或注入。
- 追问方向：哪些记忆即使相关，也必须等用户确认后才能升级为稳定事实？

### 11. 用户私有记忆和共享记忆怎么共存？

- 题目：在团队产品里，既有用户个人偏好，也有团队共享规则，你怎么避免两者互相覆盖？
- 面试官想听什么：你是否能处理“多层记忆同时存在”时的优先级，而不是只会说做权限隔离。
- 短答版：我会把它当成 merge policy 问题来处理，而不是单纯的隔离问题。共享记忆给默认上下文，私有记忆给个体覆盖，冲突时按预先定义的优先级合并。
- 深答版：这题重点是“共存时怎么决策”，不是“谁能写”。共享记忆通常承载团队规则、组织知识和默认配置，私有记忆承载个人偏好和个体历史，它们在一次回答里经常会同时生效。边界上，系统必须事先定义冲突合并规则，比如安全和合规约束最高，共享规则给默认值，用户偏好只在允许范围内覆盖默认值；否则模型就会在多条相关记忆之间临场摇摆。工程取舍在于，显式 merge policy 会增加实现复杂度，但能让行为可预测、可解释；如果没有这一层，系统看起来像“偶尔个性化，偶尔失忆”。面试里最好把重点放在优先级、覆盖条件和冲突回退，而不是再回到 ownership 本身。
- 追问方向：如果用户偏好和团队规范冲突，最终以谁为准？

### 12. 记忆已经被污染了，怎么清理？

- 题目：如果上线后发现系统记住了很多错的、脏的、过时的信息，你会怎么救？
- 面试官想听什么：你有没有治理闭环，还是只会在设计时谈“尽量别写错”。
- 短答版：我会分三步做：先止血，暂停高风险写入和召回；再定位污染来源和影响范围；最后批量清理、重建索引，并补上规则防止再次发生。
- 深答版：这题考的是故障恢复能力。真正的记忆系统一定要假设污染会发生，所以要有回滚和清污手段，比如按时间窗、来源、类型或版本批量失效，必要时重算摘要和重建向量索引。边界上，清理不只是删错数据，还要处理这些脏记忆派生出的摘要、缓存和排序特征，否则表面删了，行为还是不稳。工程取舍在于，做可回溯、可批量修复的治理链路会增加存储和运维成本，但这是长期系统必需的保险。面试里最好补一句，污染清理必须和写入审计、数据 lineage 配套，否则你连脏东西是怎么进去的都查不清。
- 追问方向：如果不能停机，你会怎么做在线清理和灰度修复？
