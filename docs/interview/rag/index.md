# RAG

做过知识型 Agent 的岗位，几乎一定会问 RAG。真正拉开差距的不是背概念，而是你能不能定位问题到底出在检索层还是生成层。短答版适合快速口述，深答版则专门补故障定位、指标和系统取舍。

## 高频题

### 1. RAG 为什么会答不准？

- 题目：如果文档明明在库里，为什么模型还是答错？
- 面试官想听什么：你是否具备定位问题的能力，而不是把一切失败都归因给“模型幻觉”。
- 短答版：我一般会分四层看：没召回到、召回到了但排序靠后、注入了但上下文被噪声淹没、模型读到了但没按证据回答。很多 RAG 问题不是模型本身不行，而是检索链路失真。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的本质是建立 RAG 的故障分层视角。RAG 答不准，通常不是一个单点问题，而是检索、排序、注入和生成四层中任意一层出了偏差。边界上，如果文档根本没召回到，那是检索问题；如果召回了但模型没用好，才更像生成问题，所以定位顺序非常关键。工程取舍在于，只有先定位到层级，你才能决定是调 embedding、加 rerank、改 chunk，还是改 prompt 和引用约束。常见误区是直接换更强模型，但如果检索结果本来就不对，再强的模型也只是在高质量地读错材料。
  </details>
- 追问方向：你会怎样一步步定位是哪一层出了问题？

### 2. 混合检索为什么常常优于纯向量检索？

- 题目：只做 embedding 检索为什么不一定够？
- 面试官想听什么：你是否理解不同检索方式的优势边界，而不是把向量检索当成万能答案。
- 短答版：纯向量检索擅长语义相似，但对精确关键词、实体名、编号和结构化约束不够稳。混合检索结合关键词和语义检索，通常更能兼顾 recall 和 precision。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的核心判断是，检索问题往往不是单一相似度函数能完全覆盖的。向量检索对语义近似很强，但对专有名词、产品编号、错误码、日期等精确匹配信号不够敏感；关键词检索正好补这块短板。边界上，混合检索不是简单把两套结果拼起来，而是利用不同检索机制覆盖不同信号，再统一排序。工程取舍在于，混合检索会增加实现复杂度和排序成本，但在企业知识库、代码、FAQ 这些混合语义与结构信号的场景里，收益通常非常明显。常见误区是迷信 embedding，结果系统在“关键词极强但语义弱”的查询上连续失手。
  </details>
- 追问方向：什么时候纯关键词检索反而更靠谱？

### 3. RAG 的核心指标该看什么？

- 题目：如果你要评估一个 RAG 系统，重点看哪些指标？
- 面试官想听什么：你是否知道 RAG 评估必须分层，否则出了问题根本不知道该改哪。
- 短答版：我至少会分成两层：召回质量和最终答案质量。前者看命中率、排序质量、上下文相关性，后者看答案正确率、证据一致性和是否真的用到了有效材料。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题本质是在看你有没有可操作的评估框架。只看最终答案是否“像对的”，不足以判断是检索差、注入差还是生成差，所以至少要把检索层和回答层拆开看。边界上，检索层更关注有没有把对的材料放进候选集合，回答层则关注模型是否基于这些材料得出正确且可引用的结论。工程取舍在于，分层评估会增加标注和构造数据集的成本，但能显著提升系统调优效率，因为你知道问题出在哪一层。常见误区是线上指标不好就同时乱改检索和 prompt，最后既看不出因果，也无法稳定迭代。
  </details>
- 追问方向：为什么离线指标和线上用户满意度常常不完全一致？

### 4. 上下文注入是不是越多越好？

- 题目：既然都召回出来了，为什么不全给模型？
- 面试官想听什么：你是否有上下文预算意识，知道召回结果进入模型前还需要二次治理。
- 短答版：不是越多越好。过多上下文会稀释关键信息、抬高成本，还可能引入互相冲突的片段。更常见的做法是控制 chunk 质量、重排、去重后再送最相关的一小部分。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的核心是“召回结果不等于最终上下文”。模型的注意力和上下文窗口都是有限资源，如果把所有召回片段都塞进去，很容易让关键证据被噪声淹没。边界上，注入阶段的任务不是追求数量，而是追求“足够支撑回答的最小证据集合”。工程取舍在于，做重排、去重和摘要会增加一点前处理成本，但它通常比盲目增加 token 更稳、更省钱。常见误区是因为担心漏信息就把更多 chunk 丢给模型，结果答案反而更飘。
  </details>
- 追问方向：chunk 太大和太小分别会带来什么问题？

### 5. RAG 和 Fine-tuning 怎么选？

- 题目：如果业务方说“知识答不准，要不要微调模型”，你怎么判断？
- 面试官想听什么：你是否知道“知识问题”和“行为问题”通常要用不同手段解决。
- 短答版：如果问题是外部知识更新频繁、需要引用原始资料，优先考虑 RAG；如果问题是输出风格、任务格式或固定行为模式不稳定，微调可能更合适。很多项目不是二选一，而是分层组合。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的本质是区分“模型不知道”还是“模型不会按你想要的方式表达”。RAG 更适合解决知识接入和知识更新问题，因为外部资料可以独立维护并保持可引用；微调更适合解决格式稳定、行为偏好和任务模式一致性问题。边界上，如果知识变化很快或者要求明确引用来源，优先微调通常不划算，因为你会陷入频繁训练和过时知识的维护成本。工程取舍在于，RAG 的检索链路更复杂，微调的训练和版本治理更重，实际系统常常是“RAG 负责知识，prompt/微调负责行为”。常见误区是只要答错就提议微调，但很多时候真正的问题只是 chunk、召回或 prompt 没做好。
  </details>
- 追问方向：什么迹象说明问题根本不在模型参数，而在检索链路？

## 扩展题

### 6. chunk 应该怎么切，为什么这不是纯参数题？

- 题目：RAG 里的 chunk size 和 overlap 应该怎么定？
- 面试官想听什么：你是否理解 chunk 设计影响的是召回边界、证据完整性和噪声比例，而不是只会报几个经验数值。
- 短答版：chunk 太大，主题混杂、排序变差；chunk 太小，语义断裂、证据不完整。我的判断标准不是固定字数，而是看一个 chunk 能不能尽量表达一个相对完整的语义单元。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的关键是把 chunk 看成检索系统的最小证据单位，而不是简单切文本。边界上，切得太大会把多个主题绑在一起，embedding 容易平均化，召回命中了也未必是你想要的那段；切得太小又会让上下文断裂，模型拿到的是碎片证据，回答时容易拼错。overlap 也不是越大越好，过大意味着索引膨胀、重复召回和 rerank 浪费。工程取舍通常要结合文档结构看，比如 FAQ、API 文档、规章制度、长报告都不该用同一套切法。常见误区是先拍一个 500/100 的参数就长期不动，但真实系统更应该按文档类型和失败样本去反推 chunk 策略。
  </details>
- 追问方向：你会优先按字数切、按标题切，还是按语义段落切？

### 7. 明明已经召回了，为什么还要加 reranker？

- 题目：有了向量检索或混合检索，reranker 的价值在哪里？
- 面试官想听什么：你是否知道召回和排序是两件事，能解释 reranker 解决的不是“找不到”，而是“排不准”。
- 短答版：召回阶段更关注别漏掉，reranker 负责把真正最相关的证据顶到前面。很多系统不是没找到答案，而是正确 chunk 排在后面，没进最终上下文。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题本质是在区分 recall 和 ranking。第一阶段检索通常要在速度和覆盖率之间平衡，所以会保留一批候选；但候选里常常混有语义相近却不回答问题的片段。reranker 的价值就在于，用更重但更准的相关性判断，把有限上下文预算留给最值得注入的证据。边界上，如果候选集合本身质量已经很高、上下文预算也很宽，reranker 收益会下降；但在文档冗长、chunk 很多、查询表达模糊的场景里，它通常很值。工程风险是 reranker 增加延迟和成本，所以更适合放在 topK 缩小后的第二阶段，而不是替代第一阶段检索。
  </details>
- 追问方向：什么情况下你会判断 reranker 带来的延迟不值得？

### 8. query rewrite 要做到什么程度，什么时候该停？

- 题目：用户问题要不要先改写再检索？
- 面试官想听什么：你是否理解 query rewrite 是提升召回的工具，但一旦改写过度，就会把用户真实意图改坏。
- 短答版：我会把 rewrite 限制在补全实体、消歧、展开缩写、补上下文这几类，不让它替用户改需求。检索前改写可以更稳，但边界一定要保守。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的核心不是“要不要改写”，而是“改写的权限有多大”。很多真实查询很短、带上下文省略、还夹杂代词或缩写，这时适度 rewrite 能明显提升召回；但如果改写开始引入新约束、替用户猜业务目标，风险就会从提升 recall 变成污染 intent。边界上，query rewrite 更适合做表达层修复，不适合做任务层重写。工程取舍在于，rewrite 越激进，单次命中率可能看起来变高，但长期会让系统更难解释，因为你不知道失败到底是检索差，还是重写把问题改错了。常见做法是保留原 query、改写 query 和检索日志，方便回溯。
  </details>
- 追问方向：如果 rewrite 后结果更差，你怎么证明问题出在改写而不是检索器？

### 9. RAG 为什么常被要求“必须带引用”，难点到底在哪？

- 题目：做 citation 看起来很简单，为什么落地时经常出问题？
- 面试官想听什么：你是否理解引用不是“把链接贴上去”这么简单，而是证据绑定、一致性校验和用户信任问题。
- 短答版：难点不在展示引用，而在保证回答里的每个关键结论都能对应到真实证据，而且引用片段和最终表述不能互相打架。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的关键是 citation consistency。用户要的不是“有几个参考链接”，而是答案中的关键事实能被逐条追溯。边界上，引用不一致通常出现在三种情况：模型把多个 chunk 混写成一个结论、引用标到了相关但不支持结论的片段、或者知识库里有版本冲突但系统没处理。工程取舍在于，想把引用做稳，往往要增加回答约束、片段级标注、证据对齐甚至后校验，这些都会抬高复杂度和延迟，但这是提升可信度的必要成本。常见误区是只要求“输出引用编号”，结果看起来像合规，实际证据链是断的。
  </details>
- 追问方向：如果两个来源都像对的，但结论冲突，你会怎么处理引用展示？

### 10. 知识库经常更新时，RAG 怎么保证检索新鲜度？

- 题目：业务文档每天都在变，怎么避免系统还在回答旧版本？
- 面试官想听什么：你是否意识到 RAG 不只是检索算法问题，还包括索引更新、版本管理和线上一致性。
- 短答版：关键不是“能不能重新建索引”，而是多快感知变更、多快完成更新，以及新旧版本切换时会不会混答。新鲜度是数据管道问题，不只是模型问题。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的本质是 retrieval freshness。很多团队只关注召回准确率，却忽略知识库更新链路是否可靠。边界上，如果文档更新频繁，而索引刷新慢、缓存没失效、旧 chunk 没退场，就会出现系统引用的是历史答案但看起来又很自信。工程取舍在于，实时更新最理想，但成本高、复杂度高，所以实际常在全量重建、增量更新、版本分层和热点文档优先刷新之间做平衡。另一个风险是新旧版本并存时排序不稳，用户拿到的是“部分新、部分旧”的混合答案，这比完全旧还危险，因为更难被察觉。
  </details>
- 追问方向：如果索引更新不能做到实时，你会优先保障哪类文档的新鲜度？

### 11. RAG 为什么既要做离线评估，也要盯线上指标？

- 题目：离线 eval 和线上 eval 各自解决什么问题？
- 面试官想听什么：你是否知道离线评估适合做可控对比，线上评估才反映真实流量和真实任务价值，两者不能互相替代。
- 短答版：离线 eval 适合做方案筛选和回归测试，线上 eval 才能看用户是否真的得到更好的答案。离线保证可比，线上保证有效。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的核心是理解评估场景不同，结论边界也不同。离线 eval 的优势是可重复、低成本、可分层定位，所以很适合比较 chunk 策略、检索器、reranker 和 prompt 版本；但它受数据集覆盖限制，往往对真实长尾问题不敏感。线上 eval 则能看到真实查询分布、用户停留、追问率、人工纠错率这些更接近业务价值的信号，但噪声更大、归因更难。工程取舍在于，好的 RAG 团队不会只靠一个分数拍板，而是先用离线把明显差的方案淘汰，再在线上用灰度验证真实收益。常见误区是离线涨分就直接全量上线，结果线上用户反而更不满意。
  </details>
- 追问方向：如果离线指标更好了，但线上满意度下降，你会先查什么？

### 12. 怎么区分是模型幻觉，还是检索根本没给到对的证据？

- 题目：用户说“这题答错了”，你怎么判断是 hallucination 还是 retrieval miss？
- 面试官想听什么：你是否有系统化诊断思路，能把“答错”拆成证据缺失、证据冲突、证据已给但模型没遵循三类问题。
- 短答版：我会先看正确证据有没有进候选、有没有进最终上下文，再看模型回答是否真的引用了这些证据。先证据，后模型，不然很容易误判。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题本质是在做故障归因。很多人一看到答错就说“幻觉”，但如果正确材料根本没被召回，那就不是 hallucination，而是 retrieval miss；如果正确材料进了候选却没进上下文，更像排序或注入问题；如果证据已经在上下文里，模型还编了新结论，才更接近生成层幻觉。边界上，最难的是证据不完整或彼此冲突，这时模型可能不是纯编造，而是在不完整材料上做了过度推断。工程取舍在于，诊断链路必须保留 query、候选集、最终上下文和回答日志，否则你只能凭感觉调系统。常见误区是因为大模型看起来“像在胡说”就直接换模型，结果真正的问题其实是检索链路没有把证据送到位。
  </details>
- 追问方向：如果候选里有正确答案，但最终回答用了错误证据，你会先改哪一层？
