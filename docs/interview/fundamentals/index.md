# 基础概念

这一类题最常出现在一面或开场提问里。面试官通常不是要听大词，而是看你能不能用干净的边界描述问题。下面每道题都拆成“短答版”和“深答版”，前者帮你快速组织口头表达，后者帮你应对连续追问。

## 高频题

### 1. Agent 和普通 LLM 对话有什么本质区别？

- 题目：如果都是调用模型 API，为什么一个系统叫 Agent，另一个只是聊天机器人？
- 面试官想听什么：你能不能把“模型能力”和“系统能力”分开讲，避免把 Agent 说成一个更强的大模型。
- 短答版：核心区别不是会不会对话，而是有没有“决策后行动，再根据结果继续决策”的闭环。普通 LLM 主要负责生成文本，Agent 则把模型输出转成动作，调用工具、接收反馈并继续推进任务。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的本质是区分“推理能力”与“执行框架”。LLM 本身只能在输入上下文里做下一步 token 预测，它没有文件系统、浏览器、数据库这些外部行动能力；Agent 是把模型放进一个运行时里，让它能借助工具和状态管理去完成任务。边界上最关键的是，Agent 的价值不在回答更花，而在它能够和环境发生真实交互。工程上这意味着你要设计工具层、状态层、权限层和循环控制层，而不是只换一个更强模型。常见误区是把“多轮聊天 + 长 prompt”误当成 Agent，但如果系统没有行动闭环，它通常还只是增强版聊天系统。
  </details>
- 追问方向：如果一个系统只能调用固定一个工具，还算不算 Agent？

### 2. Agent 和 Workflow 有什么区别？

- 题目：为什么不是所有自动化流程都该叫 Agent？
- 面试官想听什么：你是否理解“开放决策”和“固定编排”是两种不同系统，不会把所有自动化都包装成 Agent。
- 短答版：Workflow 的步骤是预先编排好的，优点是稳定、可审计、成本低；Agent 的价值在于路径不确定、需要模型动态决定下一步做什么。真正的边界不是复杂度，而是决策空间是否开放。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的核心判断是，Workflow 解决的是“我已经知道怎么做，只需要系统稳定执行”，Agent 解决的是“目标明确，但步骤和工具选择不能完全提前写死”。所以边界不在于流程长不长，而在于系统是不是要在运行时做选择。工程取舍上，Workflow 更可预测，出错更容易定位；Agent 更灵活，但调试、评估和成本控制都更难。很多业务场景其实是“Workflow + Agent 局部节点”，例如在固定审批流里只让 Agent 处理资料归纳或异常解释。常见误区是因为 Agent 看起来更高级，就默认用 Agent 重写所有流程，结果把本来很确定的事务逻辑变成高成本、低可控的系统。
  </details>
- 追问方向：什么场景下 Workflow 明显优于 Agent？

### 3. 一个系统至少具备什么，才值得叫 Agent？

- 题目：你会怎么定义最小 Agent？
- 面试官想听什么：你有没有一个最小可落地定义，而不是把记忆、规划、RAG、Multi-Agent 全部混成“标准配置”。
- 短答版：我会把最小 Agent 定义成三件事：模型负责理解和决策，工具负责执行动作，执行循环负责把结果继续喂回模型。记忆和规划很重要，但属于增强能力，不一定是最小闭环的硬前提。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题本质是在看你有没有“最小系统观”。如果一个系统连“决策 -> 执行 -> 反馈”都没有闭环，就很难说它是 Agent；反过来，只要这个闭环成立，它就已经具备最小 Agent 的核心骨架。边界上，记忆、规划、反思、RAG 都是在不同场景下增强闭环质量的能力，而不是每个 Agent 起步时都必须装满的零件。工程上先做最小闭环，再按真实问题补记忆和规划，比一开始堆满能力更稳，因为你能更清楚地知道每个模块到底解决什么问题。常见误区是把“功能越多越像 Agent”当成判断标准，结果系统一上来就过度设计。
  </details>
- 追问方向：为什么你把记忆和规划排在“可增强能力”而不是“最小必要条件”？

### 4. 什么时候不该用 Agent？

- 题目：如果业务方说“我们也想上 Agent”，你怎么判断是不是过度设计？
- 面试官想听什么：你是不是有工程克制，而不是一味鼓吹 Agent。
- 短答版：如果流程高度固定、容错空间极小、必须强审计，通常不该默认上 Agent。Agent 更适合目标明确但路径不明确的问题，不适合高风险、低容错的事务型流程。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的核心判断是“Agent 不是默认答案”。当业务流程本身非常确定，比如转账、报销审批、订单扣费、合规校验，固定逻辑和 Workflow 往往更稳，因为它们可预测、可回放、可审计。边界上，Agent 更适合信息不完整、路径需要探索、需要结合外部工具动态决策的任务，比如调研、代码辅助、复杂客服归因。工程取舍在于，Agent 会带来更高的灵活性，但同时也带来更高的不可控性、评估成本和安全风险。常见误区是把“业务方觉得新”当成技术依据，真正该问的是：这件事到底需不需要运行时的动态决策。
  </details>
- 追问方向：银行转账、发票审核、表单提交流程为什么不适合默认交给 Agent？

### 5. Agent 最大的局限是什么？

- 题目：如果你要用一句话提醒团队别神化 Agent，你会怎么说？
- 面试官想听什么：你有没有长期运行系统的视角，知道 Agent 真正难的不是 demo，而是稳定性和可控性。
- 短答版：Agent 最大的局限不是不够聪明，而是它在成本、稳定性、可控性和可验证性上天然比固定系统更难。它擅长处理不确定任务，但也更容易在边界条件下失控。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题本质上考察你是否理解 Agent 的工程代价。Agent 之所以有价值，是因为它能处理开放任务；但开放任务也意味着它更难像传统服务那样做到完全可预测、完全可审计。边界上，模型幻觉只是表层问题，真正的长期难点在于：系统会不会误选工具、会不会在异常路径上反复重试、会不会消耗过高成本、会不会做出高风险动作。工程取舍因此不只是“效果更强”，而是要配套评估、监控、权限、防护和人工接管机制。常见误区是把 Agent 看成一个会自动成长的智能员工，但实际它更像一个需要严格护栏和运行时治理的半自治系统。
  </details>
- 追问方向：你会优先从哪些工程手段降低这种不确定性？

## 扩展题

### 6. Agent 和普通自动化最大的分界线是什么？

- 题目：同样都能“自动完成任务”，Agent 和脚本自动化、RPA、定时任务的本质差别到底在哪？
- 面试官想听什么：你能不能把“自动执行”和“运行时决策”拆开，不会把一切自动化都包装成 Agent。
- 短答版：自动化强调按预设路径稳定执行，Agent 强调在运行时根据上下文决定下一步。边界不在于有没有调用 API，而在于系统是不是在真实环境反馈下持续做选择。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题最容易掉进“都能自动干活，所以都是 Agent”的坑。普通自动化通常是人先把路径写好，系统按规则执行，目标是稳定和可审计；Agent 则是在目标给定后，让模型结合环境反馈决定查什么、用什么工具、何时结束。边界上，能不能多轮运行不是关键，关键是决策权是在设计时写死，还是在运行时开放。工程取舍也很明确，自动化更稳、更便宜、更容易回放；Agent 更灵活，但也更难测、更难控。真正成熟的系统往往不是二选一，而是把确定流程留给自动化，把不确定节点交给 Agent。
  </details>
- 追问方向：如果一个流程只有最后一步需要模型判断，整体还算自动化还是 Agent？

### 7. 搜索增强或 UI 自动修复，什么时候还不算 Agent？

- 题目：带搜索的问答系统、会自动点按钮修页面的工具，为什么很多时候不该直接叫 Agent？
- 面试官想听什么：你是否能识别“能力增强”和“自治闭环”的区别，不会只看表面效果命名系统。
- 短答版：有搜索、有浏览器操作不等于就是 Agent。只要路径仍然固定，模型只是填一个步骤里的内容，它更像检索系统或自动化工具，而不是自治 Agent。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题是在考你会不会被“能查、能点、能改”这些动作迷惑。一个搜索增强问答，如果流程始终是“检索 -> 拼接上下文 -> 生成回答”，模型并没有决定是否检索、检索几次、失败后怎么改路径，那它本质上还是 RAG 问答。UI 自动修复也一样，如果规则早就写成“发现样式报错就调用修复器”，模型只负责生成补丁文本，那更接近带模型的工程工具。边界上，动作本身不稀缺，关键是系统有没有基于反馈动态调整计划。工程上乱贴 Agent 标签的问题在于，会让团队误判风险，把本该用稳定指标衡量的系统说成“智能体能力”。
  </details>
- 追问方向：一个浏览器代理如果只能按录制脚本点页面，为什么不能算真正的 Agent？

### 8. 什么样的系统看起来很聪明，但其实不是 Agent？

- 题目：有些产品能多轮对话、能总结历史、还能切换模板，为什么你还是会说它不是 Agent？
- 面试官想听什么：你能不能指出“看起来像智能”与“具备行动闭环”不是一回事。
- 短答版：多轮对话、记忆用户偏好、回答很像真人，都只能说明交互体验更强，不等于它有自主行动能力。只要系统没有围绕目标做决策和执行闭环，它大概率仍然是增强型 Chatbot。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题常见于产品 demo 很亮眼的时候。很多系统会做上下文记忆、角色扮演、语气模仿、长对话承接，所以用户主观上会觉得“它像个 Agent”；但从工程定义看，这些能力大多还停留在输入输出层，没有形成“为了完成任务去调用外部能力并根据结果继续推进”的闭环。边界上，智能感来自交互体验，Agent 性来自运行机制，这两者不能混。工程取舍上，如果你把这类系统误判成 Agent，就容易引入过度复杂的评估、安全和编排设计；反过来，承认它只是高级对话系统，反而更容易把问题拆干净。
  </details>
- 追问方向：带长期记忆的客服机器人，为什么仍可能不是 Agent？

### 9. Agent 和 Copilot / Assistant 的边界怎么讲？

- 题目：很多产品都叫助手、Copilot、Agent，这几个词在面试里你会怎么区分？
- 面试官想听什么：你能不能把“辅助用户”与“代表用户执行”拆开，不会把交互形态和系统能力混为一谈。
- 短答版：Assistant 更偏回答和建议，Copilot 更偏人在回路里的协作增强，Agent 则更强调围绕目标自主决定并执行下一步。区别不在名字，而在系统到底是在“给建议”，还是“拿着反馈继续做事”。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题的关键不是背定义，而是抓责任边界。Assistant 通常以对话和建议为主，输出价值主要停留在信息层；Copilot 往往嵌在具体工作流里，帮助人更快完成任务，但关键决策和提交动作通常还在用户手上；Agent 则更进一步，它会根据目标、上下文和外部反馈主动选择动作并持续推进。边界上，这三类系统可以混合存在，但不能因为界面都是聊天框就混成一类。工程取舍在于，越往 Agent 走，收益是自动化更强，代价是权限、安全、回放和人工接管都要更重。面试里如果你能先用“谁负责最后一步动作”把边界讲清，后面基本不容易乱。
  </details>
- 追问方向：一个代码 Copilot 如果能自动改文件但还要人确认提交，它更像 Copilot 还是 Agent？

### 10. 判断一个需求值不值得先做成 Agent，最先看什么信号？

- 题目：面对一个新需求，你第一眼看什么，来判断它有没有必要先按 Agent 方向设计？
- 面试官想听什么：你是否有前置筛选意识，能先抓住最有信息量的信号，而不是一上来就讨论模型和工具选型。
- 短答版：我先看这个需求是不是“目标明确，但路径和信息获取方式在运行时才知道”。如果步骤一开始就能写死，它大概率不该先按 Agent 设计。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题和“什么时候不该用 Agent”不一样，它考的是你做方案筛选时先看什么。我的第一信号不是风险等级，也不是能不能接工具，而是任务路径是否天然不确定。比如用户只给出目标，系统需要边查边判断、边行动边修正，这种需求才值得优先考虑 Agent；反过来，如果输入、规则、步骤、输出都很稳定，那就该先想 Workflow 或普通服务。边界上，Agent 适合处理决策空间开放的问题，不适合拿来包装已经很清楚的流程。工程取舍在于，先抓“路径是否开放”这个信号，能最快避免误把所有智能需求都做成重系统，也能避免还没确认问题类型就开始堆记忆、规划和多工具。
  </details>
- 追问方向：如果一个需求前 80% 步骤固定，只有最后 20% 需要探索，你会怎么设计？

### 11. 为什么有工具调用，也不一定是 Agent？

- 题目：如果一个系统已经会调用搜索、数据库或浏览器工具了，为什么你还是可能不把它叫 Agent？
- 面试官想听什么：你能不能指出“会调工具”和“具备自治闭环”之间还差什么，不把 Tool Use 直接等同于 Agent。
- 短答版：因为工具调用只说明系统多了执行能力，不说明它有运行时决策闭环。一次性按固定模板调工具，更像增强型功能；只有能基于结果继续判断下一步，才更接近 Agent。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题特别高频，因为很多人会把 Tool Use 当成 Agent 的同义词。实际上，工具调用只是能力接口，真正的分界线在于系统有没有把工具结果重新纳入决策，再决定是否继续、换工具、改参数或者终止。边界上，固定流程里的单次工具调用、本质上还是“模型填空 + 程序执行”；只有当系统能围绕目标在反馈里持续调整路径，工具才变成 Agent 闭环的一部分。工程取舍也很现实，把所有 Tool Use 都叫 Agent 会高估系统自治程度，进而高估它该承担的复杂编排和安全责任；反过来，把“工具接口”和“自治运行时”拆开，系统边界会清楚很多。
  </details>
- 追问方向：如果工具链路是“检索一次 -> 回答一次”，你会怎么追问判断它是不是 Agent？

### 12. 面试里最常见的 Agent 误定义有哪些？

- 题目：如果候选人把 Agent 说错了，你最常听到哪几种说法？
- 面试官想听什么：你是否见过常见误区，并能快速纠偏到清晰定义。
- 短答版：最常见的误定义有三种：把 Agent 说成更强的模型、把 Agent 说成所有自动化、把 Agent 说成必须有记忆和规划的大而全系统。它们共同的问题是都没抓住运行时决策闭环。
- 深答版：

  <details>
  <summary>展开深答版</summary>

  这题适合用“错在哪里”来展示边界感。第一种误定义是把 Agent 当成模型能力升级版，好像 GPT 更强了就自然变成 Agent；问题在于它混淆了模型和运行时。第二种是把任何自动执行流程都叫 Agent，这又忽略了固定编排和开放决策的区别。第三种则是把 Agent 说成必须具备记忆、规划、反思、多 Agent 协作，结果把增强能力误当成最小定义。边界上，这三种说法方向不同，但都绕开了最关键的一点：Agent 的核心不是“功能多”，而是系统能否围绕目标在环境反馈里持续做决定。面试里如果你先把这条主线讲清，再补充哪些能力是常见增强项，基本就不会被问散。
  </details>
- 追问方向：如果候选人把 RAG 问答直接说成 Agent，你会怎么追问把边界问清？
