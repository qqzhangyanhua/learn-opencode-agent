# 规划

规划题常出现在“你是否做过复杂 Agent 任务”的追问里。关键不是会不会说“先拆任务”，而是知道什么时候该拆、怎么拆、拆完怎么收敛。短答版帮你快速建立主线，深答版则专门补触发条件、计划质量和成本控制。

## 高频题

### 1. 为什么 Agent 需要 Planning？

- 题目：模型不是已经可以边想边做了吗，为什么还要单独做计划？
- 面试官想听什么：你是否理解 Planning 解决的是复杂任务的全局结构问题，而不是给所有任务都加一个“看起来更高级”的步骤。
- 短答版：在任务复杂、步骤多、依赖关系明显时，只靠局部 ReAct 容易短视。Planning 的价值在于先建立全局任务结构，减少走一步看一步导致的反复试错和遗漏。
- 深答版：这题的本质是解释 Planning 解决的具体问题。ReAct 擅长局部闭环，但在长任务里容易只看见眼前一步，忽略依赖顺序、阶段目标和验证节点；Planning 则是先把任务拆成可管理的阶段。边界上，不是所有任务都需要显式计划，只有当任务跨度足够长、路径不稳定、失败代价较高时，Planning 的收益才明显。工程取舍在于，Planning 会增加额外 token 和延迟，但能换来更少的盲目试错和更清晰的执行路径。常见误区是把 Planning 理解成“先写一份漂亮提纲”，而真正有价值的计划必须能映射到后续执行动作和验证节点。
- 追问方向：什么任务明显不值得上显式 Planning？

### 2. ReAct 和 Planning 的区别是什么？

- 题目：两者看起来都在“思考后行动”，边界到底在哪？
- 面试官想听什么：你能不能说清两者的职责分层，而不是把它们当成两个互斥名词。
- 短答版：ReAct 更偏局部闭环，每一步根据当前观察继续下一步；Planning 更偏全局视角，先把任务拆成阶段，再逐步执行和更新。复杂任务里常见的做法是先 Planning，再在子任务内部用 ReAct。
- 深答版：这题的核心判断是，ReAct 和 Planning 的区别主要在“视角层级”。ReAct 解决的是当前这一步该怎么做，它强调观察、推理、行动的局部循环；Planning 解决的是整个任务该分几段、先后顺序是什么、每段怎么验收。边界上，两者并不冲突，很多系统就是用 Planning 做全局框架，再用 ReAct 推进局部执行。工程取舍在于，如果只用 ReAct，复杂任务容易迷路；如果一切都先强制 Planning，又会让简单任务变慢变重。常见误区是把两者当成二选一，而真实系统里它们更像不同粒度的控制机制。
- 追问方向：有没有场景是 ReAct 足够，但显式 Planning 反而浪费？

### 3. 一个好计划应该长什么样？

- 题目：如果让 Agent 先出计划，你怎么判断这个计划是好是坏？
- 面试官想听什么：你是否有评价计划质量的标准，而不是只看它是否“条理清楚”。
- 短答版：好计划应该目标清晰、阶段边界明确、依赖关系合理、可验证，而且能支持中途修正。坏计划通常表现为过细、过虚，或者根本不能映射回实际执行动作。
- 深答版：这题本质在看你有没有“计划质量观”。一个好计划不只是看起来整齐，而是要能回答四件事：我要达成什么、拆成哪几个阶段、每个阶段依赖什么、完成后怎么验证。边界上，计划过粗会失去指导意义，计划过细又会在执行前就消耗掉大量上下文和 token。工程取舍在于，最有价值的计划通常是“足够细到可执行、又足够粗到可调整”，让系统既能推进又不会被计划本身绑死。常见误区是鼓励模型产出一份非常长的计划，但如果后续根本不会回看或验证，那份计划只是昂贵的装饰。
- 追问方向：为什么“计划写得很漂亮”不等于“计划能执行”？

### 4. 计划执行过程中需要更新吗？

- 题目：如果执行中拿到新信息，原计划是不是就没用了？
- 面试官想听什么：你是否知道计划是运行时工件，而不是一次生成就永远正确的静态文档。
- 短答版：计划应该是可修正的，不是一次性产物。关键是控制更新粒度，只在关键假设变化、依赖失效或阶段目标偏移时重算，而不是每一步都重做全计划。
- 深答版：这题的核心判断是，计划必须允许修正，但不能无限抖动。真实任务执行中，外部信息变化、工具失败、阶段结果偏差都可能让原计划失效，这时不更新反而会让系统继续沿错误路径前进。边界上，计划更新应该发生在关键节点，而不是每一步都重新规划，否则系统会陷入“计划开销吞掉执行收益”的状态。工程取舍在于，你要在稳定推进和动态修正之间找平衡，比如只在阶段切换、关键依赖失败或目标变更时重算。常见误区是两种极端：一种是死守旧计划，另一种是稍有波动就全盘重算，二者都会让系统失控。
- 追问方向：你怎么判断“该修计划”而不是“继续执行”？

### 5. Planning 的代价是什么？

- 题目：为什么不是所有复杂任务都默认加一个计划器？
- 面试官想听什么：你是否有成本意识，知道 Planning 不是免费午餐。
- 短答版：Planning 会增加额外 token 成本、响应延迟和系统复杂度，还可能产生“看上去合理但实际无效”的幻觉计划。它适合高复杂度任务，但会给简单任务引入额外负担。
- 深答版：这题的本质是谈 Planning 的负债。显式计划会多一次甚至多次模型调用，增加上下文长度、系统延迟和调试复杂度，如果计划质量又不高，系统只是多花钱生成了一份不可靠中间产物。边界上，复杂任务确实受益于计划，但简单问题往往直接执行更快更稳，因为计划并没有提供额外信息价值。工程取舍在于，要根据任务长度、失败成本、工具数量和依赖关系决定是否触发 Planning，而不是默认所有问题都先出计划。常见误区是觉得“先计划一下总没坏处”，但在生产系统里，这种额外步骤往往直接体现在成本和响应时间上。
- 追问方向：你会用什么信号决定是否触发 Planning？

## 扩展题

### 6. 你会用什么信号触发 Planning？

- 题目：如果不是所有任务都要先做计划，那系统到底该根据什么信号决定触发？
- 面试官想听什么：你有没有可落地的触发启发式，而不是笼统说“复杂任务就触发”。
- 短答版：我会看四类信号：步骤数是否不可一眼完成、是否存在跨工具依赖、失败回滚成本高不高、目标是否需要阶段性验收。满足越多，越值得先 Planning。
- 深答版：这题重点不是下定义，而是给出稳定的触发规则。真正可用的启发式通常来自任务结构，比如需要多个工具串联、前一步结果会限制后一步、存在外部信息不确定性，或者中间结果必须验收才能继续，这些都说明只靠局部 ReAct 容易走偏。边界上，不要把“问题长”误判成“任务复杂”，有些问题描述很长，但其实一步查询就能解决。工程取舍在于，触发阈值过低会让系统变重，过高又会让复杂任务直接裸奔，所以常见做法是把工具数、依赖深度、回滚成本和不确定性组合成启发式，而不是只看字面难度。
- 追问方向：哪些信号最容易被误判，导致本来不用规划的任务被规划了？

### 7. 怎么避免过度规划？

- 题目：如果模型很喜欢先写一大段计划，你怎么防止它把时间都花在计划上？
- 面试官想听什么：你是否知道过度规划的常见表现，以及怎么从机制上收住。
- 短答版：核心是限制计划粒度和预算。计划只需要细到能指导下一阶段，不需要把所有分支都预演完；否则 token、延迟和幻觉风险都会上升。
- 深答版：这题考的是成本控制能力。过度规划常见的表现是，模型把本来两三步能验证的事情，提前扩写成十几步，还对未来分支做大量猜测，结果计划越长，和真实执行越脱节。边界上，计划不是越完整越好，因为执行环境会变化，太早写死细节反而增加返工。工程上通常会用几个手段收住：限制计划层级，只规划到阶段而不是动作；限制计划长度，只覆盖最近一段执行窗口；限制提前承诺，不要求模型把未知条件都补全。真正好的系统不是“禁止计划长”，而是让计划只承担当前必要的指导职责。
- 追问方向：如果模型总是把子步骤拆得过细，你会改 prompt、改结构，还是加预算限制？

### 8. 什么时候该重新规划，而不是继续按原计划推进？

- 题目：执行过程中遇到偏差，你怎么判断这是局部噪声，还是已经需要重做计划？
- 面试官想听什么：你能不能给出清晰的重规划检查点，而不是事后解释。
- 短答版：我会看三类触发点：关键假设被证伪、阶段目标无法验收、后续依赖被前一步结果改写。只有碰到这些结构性变化，才值得重规划。
- 深答版：这题和“计划要不要更新”不完全一样，重点在于重规划的判断标准。很多执行噪声其实只需要局部修补，比如某个工具偶发失败、某条检索结果不理想，这类问题通常不该触发全局重算；但如果关键资源拿不到、原先依赖顺序不成立、阶段产出和目标明显脱节，那继续执行只会把错误放大。边界上，真正需要重规划的是任务结构变了，而不是执行细节有波动。工程取舍在于，检查点太少会错过纠偏时机，太多又会把系统拖成“每走一步都开会”。更稳的做法是在阶段结束、关键依赖失败、验收不通过这几个节点集中判断。
- 追问方向：局部修补和全局重规划之间，有没有中间层做法？

### 9. 你怎么验证一个计划是可执行的？

- 题目：计划生成出来以后，你怎么知道它不是一份听起来对、做起来不通的假计划？
- 面试官想听什么：你是否把计划验证当成独立环节，而不是默认模型写出来就能执行。
- 短答版：我会检查三件事：每个阶段有没有明确产出、关键依赖是否能被当前工具满足、阶段完成后是否有验收信号。缺一项，这个计划就不算可执行。
- 深答版：这题考的是计划验证意识。很多失败不是执行器不行，而是计划本身没有和现实约束对齐，比如要求调用不存在的工具、假设了拿不到的数据，或者阶段之间没有可验证输出，导致执行器只能硬猜。边界上，计划验证不是要求把整个任务先跑一遍，而是先做一致性检查：目标和步骤是否对齐、步骤和资源是否对齐、阶段和验收是否对齐。工程取舍在于，验证做得太重会抵消 Planning 收益，但完全不验又容易把坏计划直接送进执行。实际系统里，轻量校验通常比追求“完美计划”更有价值，因为它至少能提前拦掉明显不可执行的方案。
- 追问方向：如果计划逻辑合理，但现有工具做不到，你会让模型改计划还是升级工具集？

### 10. Planning 和任务拆解是什么关系？

- 题目：很多人把 Planning 等同于任务拆解，这样说够不够？
- 面试官想听什么：你能不能区分“拆开”与“安排好”这两层能力。
- 短答版：任务拆解只是 Planning 的一部分。拆解回答的是“分成哪些块”，Planning 还要回答“先后顺序是什么、依赖怎么处理、每一块怎么验收”。
- 深答版：这题常用来区分你有没有把概念讲透。单纯拆解只是把大任务切小，但切小以后如果没有顺序、依赖和验收设计，执行器还是可能乱跑，所以拆解本身不等于计划。边界上，有些任务只需要分块，不需要完整计划，比如几个独立查询并列执行；但一旦块之间存在前置关系、共享资源或者合并节点，就已经进入真正的 Planning 问题。工程取舍在于，过早把两者混为一谈，会让系统把所有拆解结果都当成必须严格串行的计划，反而丢掉灵活性。更稳的说法是：拆解解决粒度，Planning 解决结构。
- 追问方向：有没有场景只做 decomposition 就够，不值得上完整 Planning？

### 11. Planning 怎么支持并行分支，而不是把所有事都串行化？

- 题目：如果任务里有几块可以并行做，Planning 应该怎么表达，才不会把系统变成单线程？
- 面试官想听什么：你是否理解计划不仅决定顺序，也决定哪些地方可以并发推进。
- 短答版：好计划不只是排顺序，还要显式标出独立分支、汇合点和共享依赖。能并行的部分就并行，真正串行的只保留在关键路径上。
- 深答版：这题在考你有没有“关键路径”意识。很多系统一做 Planning，就把任务自动写成线性步骤，结果把本来独立的信息收集、多个候选方案评估都串行化了，白白增加延迟。边界上，并行不是越多越好，因为并行分支会增加状态合并和冲突处理成本，尤其是共享上下文或共享资源时更容易出问题。工程取舍在于，只有当分支之间相互独立、合并规则明确、失败不会相互污染时，并行才真正划算。Planning 在这里的价值，是先把可并行区和必须串行的关键路径分开，而不是默认一条链跑到底。
- 追问方向：并行分支的结果不一致时，你会在计划层合并，还是交给后续决策层处理？

### 12. 什么情况下应该跳过 Planning，直接执行？

- 题目：如果你面前是一个真实线上请求，什么情况下你会明确选择不做 Planning？
- 面试官想听什么：你是否敢于在该轻的时候轻，而不是把 Planning 当成默认正确答案。
- 短答版：当任务路径短、依赖弱、失败代价低，而且可以边做边校验时，我会直接执行。此时显式 Planning 带来的额外成本，大于它提供的结构收益。
- 深答版：这题不是让你反对 Planning，而是看你有没有边界感。像单次查询、一次工具调用、短链路修正这类任务，本身就能在局部闭环里快速完成，先做计划只会增加一次中间推理和等待时间。边界上，只要任务不需要跨阶段协同、不依赖前一步长期约束、也不涉及高成本回滚，就没必要强行拉出一个计划器。工程取舍在于，跳过 Planning 并不等于放弃控制，而是把控制下沉到执行时的局部观察和即时校验。真正成熟的系统不是“到处有计划”，而是知道什么时候该重，什么时候该轻。
- 追问方向：如果一开始判断为简单任务，执行到一半才发现复杂度升高，系统该怎么平滑切回 Planning？
