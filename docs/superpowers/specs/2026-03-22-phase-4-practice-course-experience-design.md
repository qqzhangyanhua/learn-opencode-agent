# Phase 4 实践篇课程化重构设计文档

**日期**: 2026-03-22  
**范围**: 实践篇入口页、实践项目统一学习骨架、理论到实践的双向关联  
**依赖**: Phase 1 的学习 metadata，Phase 2 的首页/导航入口，Phase 3 的章节学习组件与推荐关系

---

## 背景

Phase 1 到 Phase 3 已经完成了三件关键事情：

1. 站点有了可复用的学习路径和内容索引  
2. 首页、导航和板块首页更像产品入口  
3. 一批高价值章节已经具备学习目标、前置和下一步闭环  

但实践篇仍然保留明显的“仓库目录感”：

1. **实践首页信息很多，但入口决策不够强**  
   当前 `docs/practice/index.md` 已经包含阶段分组、章节总表和运行索引，但它更像“参考大全”，不是“先做什么、为什么从这里开始”的课程入口。

2. **绝大多数实践项目页仍然是长文教程，不像课程单元**  
   23 个项目页里，只有 `P1` 和 `P10` 接入了 Phase 3 的章节学习组件。其余项目页虽然已经有 `ProjectCard`、`RunCommand` 和正文结构，但还缺少统一的学习目标、前置要求、完成标准和下一步推荐。

3. **理论与实践之间的跳转还不够稳定**  
   一部分理论页已经有 `practiceLinks`，但当前没有一个“实践项目注册表”去承接这些关系，也没有在实践页中反向提示“这章对应哪些理论判断”。

Phase 4 的目标不是重写 23 章正文，而是把实践篇升级成一套“可选择入口、可跟练、可判断是否完成、可顺着下一步继续学”的课程体验。

---

## 成功标准

这一阶段完成后，下面三件事必须为真：

1. 用户打开实践篇首页后，可以按目标或阶段快速决定从哪一个项目开始，而不是只能先读完整目录。
2. 用户打开任一实践项目页时，都能看到统一的项目摘要、前置要求、运行入口、完成判定和推荐下一步。
3. 用户可以在关键理论章节和实践项目之间来回跳转，形成“看概念 -> 做项目 -> 回来看工程判断”的闭环。

---

## 方案比较

### 方案 A：继续手写强化首页和少量项目页

在 `docs/practice/index.md` 中继续手写更多 CTA、表格和说明段；只为少数关键项目页补齐课程化区块。

优点：
- 改动量最小
- 不需要新增共享数据层

缺点：
- 仍然依赖人工维护，结构很快再次漂移
- 无法保证 23 个项目页的一致性
- 理论/实践关联没有稳定真源

### 方案 B：建立统一实践项目注册表，由组件驱动首页与项目页

新增一个实践项目元数据注册表，作为实践首页、项目页课程骨架和理论桥接组件的共同数据源；页面正文继续保留，课程化结构由组件覆盖。

优点：
- 首页、项目页、桥接关系共享同一份数据
- 更容易一次性覆盖全部 23 个项目页
- 后续 Phase 5 搜索与发现也能直接复用

缺点：
- 需要先补齐 23 个项目的结构化数据
- 需要设计兼顾入门/进阶/综合实战的统一组件

### 方案 C：直接重写实践篇全局布局

从主题布局层面对所有实践页强制注入 hero、侧栏、项目导航和底部课程区。

优点：
- 统一性最高

缺点：
- 风险过高，容易和现有 markdown 正文、目录、局部组件互相打架
- 当前阶段并不需要布局级大改

**推荐方案：B。**

理由：Phase 4 的核心是把实践篇从“文档集合”升级成“课程产品”，而不是做一轮高风险布局重构。统一注册表 + 组件驱动可以在不推翻正文的前提下，把入口、项目结构和理论桥接一次性收口。

## 设计结论

### 1. 先建立实践项目真源，再改页面

新增统一的实践项目注册表，至少覆盖这些信息：

- 项目 ID / 路径 / 标题 / 所属阶段
- 难度 / 预计时间 / 主题标签
- 学习目标
- 前置项目与相关理论
- 运行命令 / 示例文件
- 完成判定
- 推荐下一步
- 是否适合在线运行

这份数据既服务实践首页，也服务项目页组件，还负责理论页里的“对应实践”桥接。

### 2. 实践首页要从“目录页”升级成“选课页”

实践首页应该重点回答四个问题：

1. 我现在应该从哪条路线开始？
2. 每个阶段解决什么问题？
3. 我完成一个项目后会通向哪里？
4. 如果我只是想快速做一个目标，最短路径是什么？

因此首页需要新增两层结构：

- **按目标选入口**：例如“先跑通最小闭环”“先补工程能力”“直接做综合项目”
- **按阶段浏览课程**：阶段卡 + 代表项目卡 + 明确的阶段目标和完成收益

现有的总表和运行索引保留，但不再占据首屏主导位置。

### 3. 每个实践项目页都要变成“课程单元”

实践项目页统一补齐两层结构：

1. **页首项目导览**  
   展示当前项目在整条实践路径中的位置、学习目标、前置要求、推荐运行模式、入口命令和示例文件。

2. **页尾完成闭环**  
   展示完成判定、下一步项目、回看的理论章节，以及“做完这一章应该继续去哪”。

正文里的代码讲解、运行结果、FAQ 保持不动。

### 4. 理论到实践采用“关键章节桥接”，不全书硬铺

当前最有价值的桥接点已经比较明确：

- `00-what-is-ai-agent` -> `P1`
- `03-tool-system` -> `P1`
- `04-session-management` -> `P2` / `P10`
- `intermediate/27-planning-mechanism` -> `P11`

Phase 4 不追求一次性改完所有理论页，只强化这些关键桥接页，并让实践项目页反向展示“建议先回看哪些理论章”，形成双向关系。

## Phase 4 拆分

### 04-01 实践篇入口页与课程浏览重构

目标：
- 建立统一实践项目注册表
- 把实践首页改成按目标和阶段浏览的课程入口

### 04-02 实践项目页统一学习骨架

目标：
- 建立项目导览与完成闭环组件
- 让 23 个实践项目页都具备统一课程结构

### 04-03 理论与实践双向桥接

目标：
- 在关键理论页强化“看完就练”的实践入口
- 在实践项目页中回链对应理论和下一条推荐路线

---

## 风险与约束

1. Phase 4 不做登录、真实进度系统、AI 助教、题库评分。
2. 不重写 23 个项目页正文，只在结构层补齐课程化骨架。
3. 理论桥接只覆盖关键页，不提前扩大成全书改造。
4. 数据真源应尽量集中，避免首页、项目页、桥接页各自维护一份配置。

---

## 验证方式

自动验证：
- `bun run build:strict`

人工验证：
- 打开 `docs/practice/index.md` 对应页面，确认首屏存在“按目标开始”和“按阶段浏览”两层入口
- 打开任意入门、进阶、综合项目页，确认顶部存在统一项目导览，底部存在完成判定与下一步
- 打开关键理论页，确认能直接看到对应实践项目，而不是只剩文字建议
- 检查移动端下实践首页卡片、项目导览卡和底部行动卡的折行与层级是否稳定
