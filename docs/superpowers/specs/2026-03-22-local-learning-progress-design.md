# 本地学习进度 MVP 设计文档

**日期**: 2026-03-22  
**范围**: 本地学习状态记录、章节页与实践页进度控件、内容级互斥状态、本地存储与降级处理  
**依赖**: Phase 1 的 `contentId` 与 metadata，Phase 3 的 `ChapterLearningGuide` / `ChapterActionPanel`，Phase 4 的 `PracticeProjectGuide` / `PracticeProjectActionPanel`，Phase 5 的统一发现与内容索引体系

---

## 背景

到 Phase 5 为止，这个学习站已经完成了三件关键事情：

1. 用户可以快速判断从哪里开始学  
2. 用户在关键章节和实践项目中可以看到明确的目标、前置和下一步  
3. 全站已经具备统一的学习路径、发现入口和推荐关系  

但现在还缺少一层非常轻、却很关键的“学习状态记录”：

1. **用户可以找到内容，但还不能在站内标记自己和这页的关系**  
   现在用户能判断“这页值不值得看”，却不能直接在页面里表达“我稍后再看”“这是我当前正在推进的页”或“这页我已经完成了”。

2. **站点像课程产品，但还没有最基本的本地进度感**  
   即使不做登录和云同步，课程型站点也应该至少支持最小的本地状态记录，否则用户每次回来都只能重新凭记忆判断自己学到哪。

3. **后续的继续学习、稍后阅读、路线进度缺少底层状态基础**  
   下一阶段如果要做“继续学习”卡片、本地进度汇总或路径完成度，就必须先有统一而稳定的内容级状态模型。

这次要做的不是完整学习系统，更不是账号化进度平台，而是一个足够稳、足够轻的本地学习进度 MVP：让用户可以在章节页和实践项目页里手动标记当前状态，并把它保存在本地。

---

## 成功标准

这一阶段完成后，下面四件事必须为真：

1. 用户在关键章节页和实践项目页中可以看到统一的进度控件。
2. 用户可以手动把某个内容标记为 `稍后再看`、`从这里继续` 或 `已完成`。
3. 同一内容的三种状态互斥，刷新页面后状态仍然保留在本地。
4. 本地存储异常或不可用时，页面主体阅读和学习组件仍然正常工作。

---

## 方案比较

### 方案 A：极简独立状态条

在每个章节页和实践页顶部直接插入一个轻量状态条，只负责切换 `稍后再看 / 从这里继续 / 已完成`。

优点：
- 实现最快
- UI 干扰最小
- 数据模型也最简单

缺点：
- 容易和现有课程化学习组件脱节
- 看起来像额外贴上去的工具条，不像课程产品原生的一部分
- 后续扩展时复用边界不够清晰

### 方案 B：嵌入现有学习骨架（推荐）

把统一进度控件嵌入 `ChapterLearningGuide` 和 `PracticeProjectGuide` 一层，复用现有“学习摘要 / 项目导览”的结构，让用户在进入内容页时顺手完成状态记录。

优点：
- 和现有产品化结构最一致
- 用户决策时机正确，进入页面前半屏就能完成标记
- 后续更容易扩展到继续学习卡、路径进度等上层功能

缺点：
- 需要同时改共享组件和本地存储层
- 工作量比纯状态条略高

### 方案 C：全站统一浮动进度面板

做一个浮动按钮或侧边面板，在任意内容页中都通过统一入口操作学习状态。

优点：
- 入口统一
- 后续能更容易扩成独立进度系统

缺点：
- 第一版太重
- 容易破坏阅读体验
- 和当前“内容页内最小记录”目标不匹配

**推荐方案：B。**

理由：第一版最重要的不是“功能多”，而是“像这个站原本就应该有的能力”。把进度控件嵌入 `ChapterLearningGuide` 和 `PracticeProjectGuide`，既能保持产品一致性，也为后续的本地继续学习能力打下清晰边界。

## 设计结论

### 1. 进度功能只长在内容页，不先扩到首页与发现页

第一版只在两类页面中提供本地进度记录：

1. **章节页**  
   进度控件放在 `ChapterLearningGuide` 附近，用于表达“我现在打算如何处理这一章”。

2. **实践项目页**  
   进度控件放在 `PracticeProjectGuide` 附近，用于表达“我和这个项目当前是什么关系”。

第一版明确不放在：
- 首页
- `/discover`
- 顶层导航
- 学习路径页
- 阅读地图页

原因是这次先解决“记录”本身，而不是做“汇总消费层”。

### 2. 状态只做三种，且互斥

用户可设置的状态只有三种：

- `稍后再看`
- `从这里继续`
- `已完成`

这三种状态在同一内容上必须互斥，任何时刻只能存在一种。

设计取舍：
- 默认状态为“无状态”，而不是显式显示“未开始”
- 点击新状态直接覆盖旧状态
- 第一版不支持通过再次点击当前状态来取消状态

这样可以避免交互含义不稳定，也避免用户不确定“刚才到底有没有保存成功”。

### 3. 数据模型保持极简，只记录“内容当前状态”

第一版的本地记录模型只需要以下字段：

- `contentId`
- `contentType`
- `status`
- `updatedAt`

其中：
- `contentId` 直接使用现有 metadata 真源
- `contentType` 复用现有类型体系
- `status` 是内部稳定枚举
- `updatedAt` 作为后续“继续学习”“最近学习”能力的基础

内部状态建议使用：
- `saved`
- `active`
- `done`

前台文案分别映射为：
- `稍后再看`
- `从这里继续`
- `已完成`

第一版明确不记录：
- 阅读百分比
- 停留时长
- 自动访问轨迹
- 路线完成率
- 云同步相关字段

### 4. 使用统一本地存储 key，避免后续迁移困难

本地存储建议使用一个统一 key，例如：

- `ai-agent-learning-progress`

其值为 `contentId -> progressRecord` 的映射。

这样做的好处是：
- 后续聚合“从这里继续”的内容非常直接
- 后续按学习路径反推进度更容易
- 未来如果要做 schema version 或迁移，也只有一个入口

第一版不是进度平台，因此不需要复杂 schema，只要保证结构清楚、稳定、可回退即可。

### 5. 实现边界应拆为 store、控件、接入层三层

为避免把状态逻辑散写到 markdown 页面中，建议拆成三层：

1. **本地进度 store**  
   负责读取、写入、覆盖、解析和回退本地状态。

2. **统一进度控件组件**  
   负责渲染三种状态按钮、高亮当前状态、触发写入并给出轻反馈。

3. **学习组件接入层**  
   由 `ChapterLearningGuide` 和 `PracticeProjectGuide` 接入统一进度控件，而不是在文档页中散写按钮逻辑。

这样可以保证：
- 逻辑集中
- 页面接入稳定
- 后续扩展“继续学习”或本地汇总时无需重写页面骨架

### 6. 进度控件属于“进入页面时的决策”，不应挂在 ActionPanel

第一版不建议把进度控件放在 `ChapterActionPanel` 或 `PracticeProjectActionPanel` 中。

理由：
- `ActionPanel` 更偏“读完以后做什么”
- 进度控件更偏“我现在要怎么处理这页”
- 用户进入页面前半屏时就应该能完成标记，不应强依赖滚到页尾

所以它应属于 guide 层，而不是 action 层。

### 7. UI 反馈要轻，不做全站干扰

第一版的交互反馈只需要做到：

- 当前状态按钮高亮
- 切换后有轻量“已保存到本地”反馈
- 页面刷新后能看到持久状态

第一版不需要：
- 全站 toast 系统
- 弹窗确认
- 进度侧边栏
- 顶栏状态提醒

目标是让这套功能看起来像自然嵌在学习体验中，而不是一个新工具系统。

### 8. 降级与错误处理必须优先保证阅读不受影响

第一版必须显式处理以下异常情况：

1. **缺少合法 `contentId`**  
   不显示进度控件，不写入匿名状态。

2. **`localStorage` 不可用**  
   进度能力应优雅降级，不能让页面报错或阻塞正文渲染。

3. **本地 JSON 数据损坏**  
   解析失败时直接回退为空状态，不应卡死页面。

4. **未来 schema 变化**  
   第一版只需保留最小回退能力，不做复杂迁移。

核心原则只有一个：进度功能出问题时，学习站仍然应该像没有这层功能一样正常可读。

---

## 第一版明确不做

为了避免范围膨胀，这一阶段明确不包含：

1. 用户登录与跨设备同步
2. 首页“继续学习”模块
3. `/discover`、学习路径页、阅读地图页的进度汇总
4. 自动记录阅读历史或最近访问
5. 路线完成百分比、已完成数量、学习仪表盘
6. 激励机制，如徽章、连续学习天数、排行榜
7. 推荐算法调整

这次只做“内容页内本地手动记录状态”的最小闭环。

---

## 验证方式

自动验证：
- `bun run build:strict`
- 新增 progress 相关校验脚本，至少验证：
  - 统一 progress store 存在
  - 统一进度控件组件存在
  - `ChapterLearningGuide` 已接入进度控件
  - `PracticeProjectGuide` 已接入进度控件
  - `build:strict` 已接入 progress 校验

人工验证：
- 打开一个关键章节页，切换状态后刷新页面，确认状态仍然存在
- 打开一个实践项目页，切换到另一个状态，确认状态覆盖生效
- 在不同页面分别设置状态，确认各页状态互不串扰
- 人为清空或破坏本地存储后，确认页面仍然正常显示
- 检查移动端下状态按钮换行、选中态和反馈文案是否稳定

## 下一步

基于本设计，拆成一个新的本地进度 phase，建议拆分为两部分：

- `P1`：本地进度 store 与统一状态控件
- `P2`：章节页与实践页接入、本地验证与降级处理
