# Agent 面试题专栏扩写设计

## 背景

当前仓库已经有 `docs/interview/` 这一组面向 Agent 学习者与面试准备者的内容页，整体阅读体验已经比较清晰：

- 题目先露出
- 先给短答版
- 深答版默认折叠
- 维持栏目化阅读，而不是散乱堆题

本次扩写目标不是重做信息架构，而是在保留现有栏目结构的前提下，把题库向“智能体开发人员”进一步推进，补足更贴近真实工程的题目。

## 目标

本次扩写要达成 4 个目标：

1. 保留现有 `fundamentals / planning / rag / engineering` 等分区，不新增一级栏目。
2. 新增题目继续使用“题目 + 短答版 + 深答版（默认折叠）”的页面结构。
3. 首批内容优先覆盖模型边界、执行设计、向量库生命周期、工程化落地这 4 类高频面试主题。
4. 让内容既适合面试复习，也能对真实 Agent 工程设计提供可迁移的方法论。

## 非目标

本次不做下面这些事情：

1. 不重构 `docs/interview/` 的导航结构。
2. 不新建“高级专区”或“深水区”作为新的一级栏目。
3. 不把页面改成纯专栏散文式长文，仍以面试题阅读模式为主。
4. 不一次性扩写所有栏目，先按推荐顺序逐步补齐。

## 信息架构策略

采用“保留现有分区，增强分区内部问题簇”的方案。

具体落点如下：

- `fundamentals`
  负责承接模型本体能力边界、能力错觉、上下文与推理边界等问题。
- `planning`
  负责承接任务拆解、一步执行设计、计划与执行切换、可中断与可恢复执行等问题。
- `rag`
  负责承接向量数据库更新、删除、重建、版本管理、增量与全量索引策略等问题。
- `engineering`
  负责承接观测、回滚、灰度、审计、生产稳定性等问题。

这样做的原因是：

1. 对现有站点改动最小。
2. 读者认知成本最低。
3. 后续如果某一类题目增长过快，再单独拆栏目也更自然。

## 首批选题范围

首批建议先写 14 道题，但本轮实现按推荐顺序先落前 5 道。

### 栏目候选题池

`fundamentals`

1. LLM 到底擅长什么，不擅长什么？
2. 为什么说 LLM 有能力上限，也有能力错觉？
3. 上下文窗口变长，为什么不等于 Agent 一定更强？
4. 为什么很多看起来像“推理”的能力，本质上仍然依赖上下文组织？

`planning`

1. Agent 任务的一步执行应该怎么设计？
2. 什么时候应该先规划再执行，什么时候应该直接单步反应？
3. 如何把 Agent 的执行过程设计成可中断、可恢复、可重试？
4. 为什么任务拆解太粗会失控，太细又会拖慢系统？

`rag`

1. 向量数据库的更新、删除、版本管理应该怎么做？
2. RAG 为什么不是“建一次索引就完事”？
3. 增量更新、全量重建、软删除分别适合什么场景？

`engineering`

1. 一个 Agent 系统要上线，最少需要哪些工程化能力？
2. 为什么很多 Agent Demo 能跑，但一上生产就不稳定？
3. Agent 系统怎么做观测、灰度、回滚和审计？

## 首批落地顺序

本轮按以下顺序落地：

1. `planning`：Agent 任务的一步执行应该怎么设计？
2. `planning`：如何把 Agent 的执行过程设计成可中断、可恢复、可重试？
3. `rag`：向量数据库的更新、删除、版本管理应该怎么做？
4. `fundamentals`：LLM 到底擅长什么，不擅长什么？
5. `fundamentals`：上下文窗口变长，为什么不等于 Agent 一定更强？

推荐这个顺序的原因：

1. 这 5 道题最贴近智能体开发岗位的真实工程判断题。
2. 它们覆盖“执行设计 -> 知识系统 -> 模型边界”三条主线，读者感知最强。
3. 写完这 5 道后，栏目分布已经开始均衡，后续再扩 `engineering` 更顺。

## 页面写法模板

每道题统一遵循当前 interview 页面风格，但内容密度要明显高于基础入门题。

### 页面结构

每个题目建议使用以下稳定骨架：

1. 题目
2. 短答版
3. 深答版（`<details>` 默认收起）
4. 工程判断句或落地提醒
5. 延伸阅读

### 短答版规范

短答版的职责是“让用户在 30 秒内抓住判断框架”，而不是塞满背景知识。

建议控制为 3 到 5 个要点，单条不宜过长。每条回答优先覆盖：

1. 结论是什么
2. 为什么是这样
3. 工程上应该怎么处理

短答版不展开实现细节，但必须能形成明确立场，避免写成空泛定义。

### 深答版规范

深答版默认折叠，用来承载真正区分候选人深度的解释层。

建议每题深答版按下面 4 层展开：

1. 原理层
   说明这类问题背后的系统机制或模型限制。
2. 设计层
   说明工程上应该怎样拆模块、设边界、定状态。
3. 取舍层
   说明不同方案各自适用场景、成本和风险。
4. 面试表达层
   给出一句适合面试中收口的总结。

深答版可以更长，但要避免失去“答题感”。重点是帮助读者把知识组织成可复述的判断链路。

## 首批 5 道题的内容侧重点

### 1. Agent 任务的一步执行应该怎么设计？

应重点覆盖：

- 单步执行的输入、决策、动作、结果回填
- 为什么一步不能只是“调一次模型”
- 为什么要把工具调用、状态更新、错误处理视为同一执行单元的一部分
- 一步执行的边界过大或过小分别有什么代价

### 2. 如何把 Agent 的执行过程设计成可中断、可恢复、可重试？

应重点覆盖：

- 中断点设计
- 持久化状态与恢复点
- 幂等性要求
- 工具调用副作用的补偿或去重
- 重试应发生在模型层、工具层还是任务层

### 3. 向量数据库的更新、删除、版本管理应该怎么做？

应重点覆盖：

- 文档更新不是简单覆盖 embedding
- chunk、metadata、索引版本之间的关系
- 软删除、硬删除、重建索引、增量更新的适用场景
- 为什么生产 RAG 必须考虑可追溯和可回滚

### 4. LLM 到底擅长什么，不擅长什么？

应重点覆盖：

- LLM 擅长模式归纳、语言压缩、弱结构推断
- 不擅长稳定状态跟踪、强确定性执行、长期一致性维护
- 为什么“看起来会”和“稳定能做”是两回事
- Agent 设计为什么要把模型能力与系统能力分开

### 5. 上下文窗口变长，为什么不等于 Agent 一定更强？

应重点覆盖：

- 上下文更长只意味着可放更多信息，不意味着模型能稳定利用全部信息
- 检索质量、上下文组织、状态压缩仍然决定上限
- 长上下文带来的干扰、注意力稀释、成本与延迟问题
- 为什么工程上仍然需要摘要、检索、分层记忆

## 实施方式

本轮正文实现建议采用“直接在现有栏目页追加题目块”的方式，而不是先拆成独立详情页。

原因如下：

1. 当前 interview 栏目本身就是题库式页面，追加题目块改动最小。
2. 更容易保持用户熟悉的阅读节奏。
3. 如果未来某些题目需要独立成长文，再从栏目页抽出来也不难。

对应文件为：

- `docs/interview/planning/index.md`
- `docs/interview/rag/index.md`
- `docs/interview/fundamentals/index.md`

## 验收标准

完成首批 5 道题后，应满足以下标准：

1. 每道题都符合“题目 + 短答版 + 深答版（默认收缩）”结构。
2. 短答版能在 30 秒内读完，并给出明确判断。
3. 深答版不是重复短答，而是补足原理、设计、取舍和面试表达。
4. 新增内容语言风格与现有 interview 页面一致，面向中文读者。
5. 页面结构不破坏现有导航与栏目节奏。

## 风险与约束

### 风险 1：写成泛科普

如果只讲概念，不讲工程边界，内容会偏离“智能体开发人员”的受众。

约束方式：

- 每题都必须出现工程判断句
- 深答版必须包含设计取舍

### 风险 2：写成纯长文

如果深答版失控，题库会失去“刷题感”和可扫描性。

约束方式：

- 短答版保持高密度
- 深答版只在折叠区展开

### 风险 3：分类漂移

某些题会同时落在多个栏目，例如 LLM 边界会影响 planning 与 engineering。

约束方式：

- 以“读者最可能去哪里找这类题”为第一分类原则
- 必要时在延伸阅读中互相链接，而不是重复搬运
