# 实践篇完整源码折叠展示设计文档

**日期**: 2026-03-23  
**范围**: `docs/practice/*` 章节底部完整示例源码展示、默认折叠、多文件独立面板、复制代码  
**依赖**: `.vitepress/theme/data/practice-projects.ts`、`.vitepress/theme/index.ts`、实践页现有 `project-id` 挂载方式

---

## 背景

当前实践篇正文已经包含分段讲解代码，但用户如果想直接查看并复制该章对应的完整可运行示例，仍需要离开文章去仓库里的 `practice/*.ts` 手动查找。这个行为有三个明显问题：

1. **阅读链路被打断**  
   用户在阅读完讲解后，还要切换到文件树或仓库页面，学习链路中断。

2. **正文代码与完整示例之间缺少收口**  
   文章中的代码块用于分步教学，完整源码用于最终对照与复制，两者职责不同，但当前页面没有把完整源码稳定地放到文章底部。

3. **现有元数据还没有被真正利用**  
   实践项目数据里已经有 `sourceFiles`，例如 P2 对应 `practice/p02-multi-turn.ts`，P14 对应两个源码文件，但页面层还没有消费这部分数据。

本次设计只解决一个问题：让实践篇每篇文章底部都能展示对应项目的完整源码，默认折叠，用户可展开并复制。

---

## 成功标准

这一能力完成后，下面几点必须成立：

1. 任意实践章节底部都能看到“完整示例源码”区域。
2. 每个源码文件对应一个独立折叠面板，默认折叠。
3. 用户展开后可以看到完整文件内容，并一键复制。
4. 多文件章节按 `sourceFiles` 顺序展示多个独立面板。
5. 页面不依赖运行时读取本地文件，不新增额外网络请求。

---

## 方案比较

### 方案 A：在每篇 Markdown 里手写源码引入

在每个 `docs/practice/*/index.md` 中手写 `<<< ../../practice/*.ts` 或手动插入代码块。

优点：
- 上手快
- 不需要新增组件

缺点：
- 每一章都要重复维护文件路径
- 多文件章节更容易遗漏或顺序漂移
- 不能复用现有 `sourceFiles` 元数据

### 方案 B：新增独立组件，读取 `project-id -> sourceFiles`

新增 `PracticeProjectSourceFiles.vue`，由组件根据 `project-id` 自动拿到源码路径，再统一渲染底部折叠面板。

优点：
- 最大化复用现有实践项目数据
- 组件职责清晰，只负责完整源码展示
- 单文件和多文件章节都能统一处理

缺点：
- 需要增加一个主题组件和一个源码映射数据模块

### 方案 C：在主题层自动识别 practice 页面并隐式注入

根据当前路由自动判断是否是实践页，并在主题布局底部统一插入源码区域。

优点：
- 理论上减少 Markdown 改动

缺点：
- 页面行为变隐式，定位问题更困难
- “只对显式挂载了实践项目元数据的页面生效”这一边界不够清晰
- 不利于后续局部调整页面底部结构

**推荐方案：B。**

理由：需求本质不是“魔法注入”，而是“利用已有实践元数据补齐底部源码收口”。方案 B 改动集中、边界清晰、后续也最容易维护。

---

## 设计结论

### 1. 新增独立底部组件 `PracticeProjectSourceFiles`

新增一个只负责完整源码展示的组件：

- 输入：`project-id`
- 输出：当前项目全部 `sourceFiles` 的折叠面板列表

职责边界：

- 它不负责项目简介
- 不负责运行命令
- 不负责过关标准
- 只负责“展示完整源码并允许复制”

这样可以保持当前页面结构稳定：

1. `PracticeProjectGuide` 负责顶部导学
2. `PracticeProjectSourceFiles` 负责底部完整源码
3. `PracticeProjectActionPanel` 负责底部过关信号

### 2. 源码内容在构建时导入，不在运行时读文件

新增主题数据模块，例如 `practice-source-files.ts`，将 `practice/*.ts` 作为文本在构建时导入，并建立 `path -> code` 映射。

推荐数据流：

1. `practice-projects.ts` 提供当前章节的 `sourceFiles`
2. `practice-source-files.ts` 提供对应路径的完整源码文本
3. `PracticeProjectSourceFiles.vue` 组合两者并渲染

选择构建时导入而不是运行时读取的原因：

- 更适合 VitePress 的静态构建模式
- 不依赖浏览器端额外请求
- SSR 和生产构建更稳定
- 复制按钮可直接使用已加载文本

### 3. 多文件章节按文件维度拆成多个独立折叠面板

用户已经明确要求：**每个文件一个独立折叠面板**。

因此组件需要按 `sourceFiles` 顺序逐个渲染，例如：

- P2：1 个面板
- P14：2 个面板，分别对应 `practice/p14-mcp.ts` 和 `practice/p14-mcp-server.ts`

每个面板至少包含：

- 文件路径标题
- 默认折叠状态
- 展开后的完整代码块
- 复制代码按钮
- 复制成功后的短暂反馈

### 4. 挂载位置固定在实践页底部的行动面板之前

实践页底部当前已经有：

- `StarCTA`（部分页面）
- `PracticeProjectActionPanel`

新组件应插入到 `PracticeProjectActionPanel` 之前，形成稳定顺序：

1. 正文
2. `StarCTA`（若该页存在）
3. `PracticeProjectSourceFiles`
4. `PracticeProjectActionPanel`

这样能保证用户先学正文，再看完整源码，最后看过关标准，信息顺序自然。

### 5. 样式采用文档附录风格，而不是营销卡片风格

视觉设计应克制，贴近技术文档：

- 区块标题使用“完整示例源码”
- 每个文件使用 `details / summary` 风格折叠容器
- 展开后直接展示代码块，不再嵌套复杂说明卡
- 优先复用现有 VitePress 变量和站点色板

目标不是让它成为视觉主角，而是成为文章底部稳定、可靠、好复制的工程附录。

## 组件与数据职责

### `practice-projects.ts`

继续作为实践项目元数据源，保留已有 `sourceFiles` 字段，不新增新的源码路径配置协议。

### `practice-source-files.ts`

新增源码文本映射模块，负责：

- 收集 `practice/*.ts`
- 生成 `Record<string, string>` 形式的源码内容映射
- 暴露按路径读取源码的辅助方法

### `PracticeProjectSourceFiles.vue`

负责：

- 根据 `project-id` 获取项目
- 遍历项目的 `sourceFiles`
- 读取每个文件的源码文本
- 渲染多个折叠面板
- 处理复制状态反馈

### `theme/index.ts`

负责全局注册 `PracticeProjectSourceFiles` 组件。

### `docs/practice/*/index.md`

负责在底部显式挂载组件，而不是使用主题级隐式注入。

## 异常与回退策略

### 1. 项目不存在

如果传入的 `project-id` 无法匹配到项目，组件直接不渲染，不让页面报错。

### 2. 元数据声明了源码文件，但映射中不存在

组件不应抛出异常中断页面，而应显示轻量提示，例如“源码暂未收录”，并保留该文件面板标题，方便后续排查配置问题。

### 3. 复制失败

如果浏览器复制 API 调用失败，按钮状态应恢复，并显示简短失败提示或降级文案，但不影响源码展示。

### 4. 文件过长

当前阶段不做分段懒加载，也不做折行摘要。默认假设实践文件长度仍在文档页可接受范围内；如后续出现超长文件，再单独规划滚动区域或虚拟化，而不是提前过度设计。

---

## 范围边界

本次设计明确不做以下内容：

1. 不自动汇总正文中的 Markdown 代码块。
2. 不提取交互 Demo 组件内部源码。
3. 不做主题布局级自动注入。
4. 不支持运行时从仓库或远程地址拉取源码。
5. 不增加下载 ZIP、切换主题、行高亮定位等扩展功能。

这样可以把范围稳定在“实践篇完整源码附录”这一件事上。

## 验证方式

自动验证：

- `npm run typecheck`
- `pnpm -s run build`

人工验证至少覆盖：

1. **单文件章节**  
   打开 `P2`，确认底部出现一个折叠面板，默认折叠，展开后能看到 `practice/p02-multi-turn.ts` 完整内容并成功复制。

2. **多文件章节**  
   打开 `P14`，确认底部出现两个独立折叠面板，顺序与 `sourceFiles` 一致，两个按钮互不干扰。

3. **视觉与交互**  
   确认桌面端和移动端下，折叠标题、代码块滚动、复制按钮都没有遮挡或错位。

4. **异常回退**  
   对一个缺失映射的文件进行临时验证时，页面应显示降级提示，而不是报错白屏。
