# 第28章上下文工程四阶段模拟器设计稿

## 背景

第 28 章当前已经接入 `ContextEngineeringExtended`，但它更像一个“候选上下文选择器”：

- 能看见候选项
- 能按类型过滤
- 能按相关度/最近/Token 排序
- 能看预算占用

这对理解“选哪些内容进上下文”有帮助，但它还没有把第 28 章真正的主线讲完整。因为本章真正要建立的心智模型，不是“选中几项候选内容”，而是四个连续动作：

- 选：哪些内容值得进入窗口
- 排：进入后如何重新排序
- 压：放不下时怎么保留决策所需信息
- 拼：最后如何组装成消息包

如果没有把这四步变成可切换的状态演示，读者仍然容易把上下文工程误解成“多塞一点相关内容”。

## 目标

把现有 `ContextEngineeringExtended` 升级为第 28 章的四阶段模拟器，让读者能沿着 `选 -> 排 -> 压 -> 拼` 一步步理解上下文工程的动作链，而不是只看到一个静态候选列表。

成功标准：

- 组件顶部有四阶段切换：`选 / 排 / 压 / 拼`
- 切换阶段时，中间主视图区和右侧说明都会变化
- `选` 阶段重点显示候选筛选
- `排` 阶段重点显示顺序重排
- `压` 阶段重点显示原文 / 摘要 / 关键事实三层压缩
- `拼` 阶段重点显示最终 `messages` 组装结果
- 整体仍沿用当前组件的数据，不另造完全不同的使用方式

## 范围

本次只改第 28 章及其现有组件，不扩展到 29 或其他章节。

包含：

- 升级 `ContextEngineeringExtended.vue`
- 如有必要，为第 28 章补一个轻量数据辅助文件
- 在章节体验校验里补回归检查

不包含：

- 不新建一个与 `ContextEngineeringExtended` 平行的大型新组件
- 不引入重动画库
- 不重写整章正文结构

## 方案选择

### 方案 A：升级现有 `ContextEngineeringExtended`

做法：

- 保留现有候选上下文数据结构
- 在组件中增加阶段状态
- 让同一份候选数据在不同阶段以不同视图呈现

优点：

- 与当前章节内容衔接自然
- 代码复用率高
- 改动范围可控

缺点：

- 需要在现有组件内部补更多阶段逻辑

### 方案 B：新增 `ContextAssemblySimulator`

做法：

- 新建章节级组件
- 将现有 `ContextEngineeringExtended` 保留为底层选择器

优点：

- 组件职责分层更清楚

缺点：

- 与现有组件功能重叠明显
- 对第 28 章来说成本偏高

### 推荐方案

采用方案 A。

原因：

- 当前组件已经具备候选、预算、选择等关键基础
- 第 28 章最缺的不是全新视觉容器，而是“四阶段链路”的显式状态切换
- 在现有组件上升级，最容易保持术语、数据和正文一致

## 交互设计

### 顶部阶段条

新增四个按钮：

- 选
- 排
- 压
- 拼

切换阶段时：

- 不重置用户已经选中的候选项
- 改变主视图区的展示方式
- 改变右侧说明区的解释文案

### 阶段一：选

目标：

- 让读者看清楚哪些内容被保留，哪些被排除

重点展示：

- 左侧候选列表
- 当前选中项
- 预算占用
- “为什么这个项值得进窗口”的判断

此阶段的解释重点：

- 不是所有相关内容都值得进窗口
- 要先过滤，再讨论后续组装

### 阶段二：排

目标：

- 展示上下文进入窗口后，顺序会影响注意力分配

重点展示：

- 已选项的排序结果
- 不同排序维度对最终顺序的影响
- 用户问题和 system prompt 在队列中的位置

此阶段的解释重点：

- 上下文不是集合，而是有顺序的消息流
- 顺序错误会让模型先注意到次要内容

### 阶段三：压

目标：

- 展示“近处保原文，中段做摘要，远处只保关键事实”的压缩逻辑

重点展示：

- 每个已选项的压缩状态
- 原文 / 摘要 / 关键事实三种形态
- 压缩后预算下降多少

此阶段的解释重点：

- 压缩不是简单截断
- 要保留继续决策所需的信息密度

### 阶段四：拼

目标：

- 展示最终上下文如何组装成 `messages`

重点展示：

- `system prompt`
- 历史对话
- 检索资料
- 工具结果
- 当前用户问题

此阶段的解释重点：

- 上下文工程最终落点是“可提交给模型的一组 messages”
- 不是想到什么塞什么，而是按预算和目的组装

## 视觉结构

保留当前组件的“两栏主结构”，但让右侧从“已选列表”升级为“阶段说明 + 结果面板”。

建议结构：

- 顶部：标题 + token 预算条 + 阶段切换按钮
- 左侧：候选项 / 已选项 / 压缩形态 / 排序结果
- 右侧：当前阶段说明 + 风险提示 + 最终组装结果预览

## 数据设计

尽量沿用当前 `ContextCandidate`。

如需补充，只加最小字段：

- `compressionPreview?: { raw: string; summary: string; fact: string }`
- 或者在组件内用本地映射表描述压缩后的呈现文本

原则：

- 不重构全站 `ContextCandidate`
- 不为了演示新增一套复杂 schema

## 组件边界

### `ContextEngineeringExtended.vue`

负责：

- 阶段切换状态
- 候选选择
- 预算统计
- 排序和阶段视图切换

不负责：

- 真实上下文压缩算法实现
- 真实消息组装逻辑执行

它是教学模拟器，不是运行时引擎。

## 章节接入方式

第 28 章正文里的现有组件入口保留，不改标签名，直接让组件升级。

优点：

- 文档改动最小
- 不需要重写章节接入方式

正文可补一句短引导：

“先依次切换‘选 / 排 / 压 / 拼’，再观察同一组候选上下文在不同阶段是怎样变化的。”

## 验证方案

至少验证：

- `bun run typecheck`
- `bun run check:chapter-experience`
- `bun run build`

章节体验校验新增要求：

- `ContextEngineeringExtended.vue` 必须包含阶段切换处理函数
- 组件源码里必须出现 `选 / 排 / 压 / 拼` 四阶段标识

## 风险与控制

### 风险 1：组件内部逻辑膨胀

控制：

- 只做四阶段视图切换
- 不引入自动播放和复杂状态机

### 风险 2：信息太多导致更难读

控制：

- 每阶段只突出一个核心动作
- 不让四个阶段同时展示所有信息

### 风险 3：演示文案和正文术语不一致

控制：

- 所有阶段名称严格与正文一致
- 解释文案直接围绕“选 / 排 / 压 / 拼”

## 预期产出

- 升级后的 `ContextEngineeringExtended.vue`
- 第 28 章一套更清晰的四阶段状态切换体验
- 回归校验，确保以后不会退回成单一候选选择器
