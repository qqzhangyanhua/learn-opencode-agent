# 第26章多智能体模式模拟器设计稿

## 背景

第 26 章当前已经接入 `MultiAgentWorkflowDetailed`，但它更像“消息流回放器”，适合展示单条协作链路，不足以帮助读者快速建立三种协作模式的整体心智模型：

- 主从模式关注“谁负责拆分、谁负责执行、风险在哪里”
- 辩论模式关注“相反观点如何形成对照，裁判如何收束”
- 流水线模式关注“输入如何层层传递，偏差如何逐级放大”

如果继续只靠正文说明，用户能理解概念，但不容易把三种模式的差异记住，更难在工程实践里快速判断“当前问题该用哪种协作结构”。

## 目标

为第 26 章新增一个章节级交互组件 `MultiAgentModeSimulator`，让读者通过模式切换和阶段切换，直观看到三种协作结构的差异、优点和风险。

成功标准：

- 读者能在一个组件里切换 `主从 / 辩论 / 流水线`
- 每种模式至少展示 4 个关键阶段，且每个阶段都能点击切换
- 中间区域不只是播放消息，而是同时展示“角色结构 + 当前阶段 + 关键产物”
- 组件在桌面和窄屏下都不乱
- 第 26 章正文接入新组件后，旧的阅读链路不受破坏

## 范围

本次只改第 26 章，不同时改 25 / 28 / 31。

包含：

- 新建 `MultiAgentModeSimulator.vue`
- 将第 26 章正文从旧消息流演示升级为章节级交互演示
- 补充章节体验校验，防止未来回退成静态展示

不包含：

- 不重写 `MultiAgentWorkflowDetailed.vue`
- 不把三种模式抽成跨章节的通用骨架
- 不引入重量级动画库

## 方案选择

### 方案 A：继续增强 `MultiAgentWorkflowDetailed`

做法：

- 在现有组件上增加模式 tab
- 根据模式替换消息列表

优点：

- 改动小，开发快

缺点：

- 组件的核心结构仍然是“消息流”
- 很难同时承载“阶段切换 + 角色结构 + 风险提示”
- 会把一个本来偏底层的组件堆成巨型组件

### 方案 B：新增章节级模拟器，保留旧组件

做法：

- 新增 `MultiAgentModeSimulator.vue`
- 保留 `MultiAgentWorkflowDetailed.vue` 作为旧组件
- 第 26 章接入新组件

优点：

- 组件职责清晰
- 交互结构更贴近教学目标
- 后续 28 / 31 可以复用“章节级模拟器”的组织方式

缺点：

- 需要新增数据结构和组件样式

### 推荐方案

采用方案 B。

原因：

- 第 26 章的问题不是“某条消息流不够丰富”，而是“整章缺少一个能承载三种模式对比的交互容器”
- 保留旧组件可以避免影响其他页面或潜在复用
- 新组件边界更干净，后续迁移和扩展成本更低

## 组件设计

### 顶层结构

`MultiAgentModeSimulator` 分成四个区域：

1. 模式切换栏
2. 场景主画布
3. 阶段说明面板
4. 底部阶段条

### 模式切换栏

提供三个按钮：

- 主从模式
- 辩论模式
- 流水线模式

切换模式时，需要同步完成：

- 重置当前阶段到该模式的第一阶段
- 切换角色卡片集合
- 切换消息流或产物流
- 切换右侧解释文案

### 场景主画布

主画布包含三部分：

- 左侧角色结构区：展示当前模式有哪些 Agent，各自职责是什么
- 中部协作过程区：展示当前阶段的消息流、观点碰撞或产物流
- 右侧阶段说明区：解释“当前阶段在这个模式里为什么关键”

三个模式共用同一骨架，但中部呈现不同：

- 主从：强调协调者到执行者的派发与回收
- 辩论：强调正反双方与裁判的来回拉扯
- 流水线：强调上游输出成为下游输入

### 阶段条

每个模式提供自己的阶段序列，底部阶段条必须可点击。

主从模式建议阶段：

1. 任务进入
2. 拆分分派
3. 子任务执行
4. 汇总审查

辩论模式建议阶段：

1. 议题提出
2. 正反陈述
3. 交叉质询
4. 裁判收束

流水线模式建议阶段：

1. 初始输入
2. 第一段加工
3. 第二段加工
4. 最终输出

### 右侧说明区

说明区固定输出三类信息：

- 当前阶段在做什么
- 这一模式的收益是什么
- 这一阶段最容易出什么问题

目标不是堆定义，而是把用户注意力拉到“结构差异”上。

## 数据设计

建议新增本地场景数据文件，例如：

- `.vitepress/theme/data/multi-agent-mode-scenarios.ts`

数据结构按“模式 -> 阶段”组织，每个模式包含：

- 模式标题
- 模式摘要
- 角色列表
- 阶段列表

每个阶段包含：

- `id`
- `label`
- `headline`
- `insight`
- `risk`
- `messages` 或 `artifacts`

这样模板只消费结构化数据，不把教学文案硬编码进组件逻辑。

## 交互原则

- 默认进入主从模式第一个阶段
- 点击模式时重置到该模式第一个阶段
- 点击阶段时只切换当前模式内部状态，不跨模式串联
- 不做复杂自动播放，优先确保用户主动切换就能理解
- 可以保留极轻的过渡动画，但不能牺牲信息稳定性

## 视觉方向

整体延续第 27 章“章节模拟器”的信息密度风格，但避免直接复制其三栏业务感。

建议视觉重点：

- 模式切换按钮要有明显“结构类型”感
- 角色卡片色彩稳定，帮助形成记忆映射
- 中部过程区允许使用轻量高亮和淡入切换
- 风险提示区和收益提示区要有明确区分

## 章节接入方式

在 `docs/intermediate/26-multi-agent-collaboration/index.md` 的“核心概念与主链路”部分，用新组件替换当前 `MultiAgentWorkflowDetailed` 入口。

文案上保留一句引导：

“先切换三种模式，再点击底部阶段，看它们在结构、风险和回流方式上的差异。”

## 测试与验证

本仓库没有专门单测框架，本次验证以静态校验和构建为主。

至少验证：

- `bun run typecheck`
- `bun run check:chapter-experience`
- `bun run build`

同时补一条章节体验校验，至少保证：

- 第 26 章已接入 `MultiAgentModeSimulator`
- 组件存在模式切换按钮
- 组件存在阶段切换处理函数

## 风险与控制

### 风险 1：组件过度复杂

控制：

- 不复用第 27 章的全部状态逻辑
- 只做“模式切换 + 阶段切换”两层状态

### 风险 2：视觉过强但信息不清晰

控制：

- 不引入重动画库
- 先保证每一屏信息稳定，再决定是否补动画

### 风险 3：组件与正文脱节

控制：

- 将模式名称、阶段名称、正文术语完全对齐
- 文案以“解释差异”而不是“重复正文”作为目标
