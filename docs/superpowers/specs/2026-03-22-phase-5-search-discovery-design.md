# Phase 5 搜索与发现体验收口设计文档

**日期**: 2026-03-22  
**范围**: 独立发现中心 `/discover`、搜索体验增强、内容类型标识统一、推荐关系收口  
**依赖**: Phase 1 的 metadata 与内容索引，Phase 2 的首页和导航入口，Phase 3 的章节学习闭环，Phase 4 的实践项目真源与理论桥接

---

## 背景

Phase 1 到 Phase 4 已经完成了学习站产品化的前四层骨架：

1. 站点已经有稳定的内容 metadata、学习路径和内容索引  
2. 首页、导航和板块入口已经具备“先分流，再进入”的产品结构  
3. 关键章节已经拥有目标、前置、练习和下一步闭环  
4. 实践篇已经从目录页升级为按阶段和目标浏览的课程入口  

但全站仍然缺少最后一层“发现收口”：

1. **用户知道这个站可以学什么，但不一定知道自己现在该从哪里开始**  
   首页、阅读地图、学习路径、实践页都已经比过去更清晰，但它们仍然是多个分散入口，缺少一个统一的“学习分诊台”。

2. **搜索能用，但还不够像课程产品**  
   当前本地搜索可以返回页面结果，但结果更像文档列表，缺少内容类型、学习语义和选择理由，用户搜到后仍要自己判断“这页适不适合我现在看”。

3. **推荐关系已经存在，但还没有统一暴露给用户**  
   Phase 3 和 Phase 4 已经积累了章节到实践、前置到下一步的推荐关系，但这些关系更多存在于单页内部，尚未形成一个面向全站发现的统一层。

Phase 5 的任务不是重写搜索引擎，也不是新增登录或真实进度系统，而是把已有的内容资产和推荐关系收口成一个更像课程平台的“发现入口 + 搜索识别 + 学习衔接”闭环。

---

## 成功标准

这一阶段完成后，下面三件事必须为真：

1. 用户可以从首页、导航或直接访问一个统一发现入口，在 30 秒内判断自己的起点。
2. 用户在搜索结果或发现页卡片中，可以清楚区分章节、实践项目、学习路线和进阶专题。
3. 用户从任意一个关键入口进入后，都能继续沿着推荐关系找到下一步，而不是回到目录重新找路。

---

## 方案比较

### 方案 A：只增强搜索结果，不新增统一发现入口

继续沿用现有首页、学习路径页和实践页结构，只针对本地搜索做结果文案、类型标签和索引字段增强。

优点：
- 变更范围最小
- 对现有导航结构冲击最小
- 能较快提升“已知关键词搜索”效率

缺点：
- 无法解决“我不知道该搜什么”的用户起步问题
- 全站入口仍然分散，学习分流依赖用户自己理解页面差异
- 课程产品感提升有限

### 方案 B：新增独立发现中心，并在现有搜索上补一层课程化识别

新增 `/discover` 作为统一发现中心，用“按目标选路线”做主轴；保留本地搜索引擎，但增强类型识别、学习语义和推荐关系展示。

优点：
- 最符合“30 秒知道从哪里开始学”的核心价值
- 可以复用现有 metadata、路径数据和实践项目真源
- 发现入口、搜索体验和推荐关系能够一次性收口

缺点：
- 需要设计新的 discovery 数据编排层
- 需要同时触达首页引流、导航入口和若干推荐组件

### 方案 C：把首页直接升级为完整发现中心

不新增 `/discover`，而是继续扩展首页，把路线选择、搜索、推荐内容和主题聚合都塞进首页。

优点：
- 减少页面数量
- 用户第一次进入后无需再跳一次页面

缺点：
- 首页会同时承担品牌、说服、分流、搜索和系统发现，职责过重
- 后续内容增加后，首页很容易重新膨胀成信息堆叠页
- 不利于导航层保持稳定心智

**推荐方案：B。**

理由：Phase 5 的核心不是继续堆首页，也不是只做一个更好用的搜索框，而是建立一个独立、明确、可扩展的“学习发现中心”。首页继续承担品牌和首屏说服，`/discover` 承担学习分流和内容发现，职责边界更清晰，也更符合后续维护。

## 设计结论

### 1. 全站入口重构为“首页引流 + 发现中心分诊 + 内容页承接”

Phase 5 之后，信息架构应稳定为四层：

1. **首页**  
   负责品牌说明、价值主张和首次引流，只保留少量强 CTA，例如“从 0 开始学”“去发现中心选路线”“直接进入实践项目”。

2. **发现中心 `/discover`**  
   作为全站统一学习起点，重点解决“我是谁、我该先学什么、下一步去哪里”的问题。

3. **路径页 / 板块页**  
   包括 `learning-paths`、`reading-map`、`practice`、`intermediate` 等，作为某种学习方式或内容集合的展开页。

4. **内容详情页**  
   具体章节、实践项目和专题页，负责完成实际学习动作。

这个结构的目标是避免让用户一进入站点就在多个板块首页之间自行判断，而是先进入统一发现中心，再由发现中心把用户导入最适合的学习路径。

### 2. `/discover` 必须先帮用户选择，再允许用户继续探索

`/discover` 的主轴不是“展示很多内容”，而是“帮助用户完成第一次选择”。页面建议分为四层：

1. **首屏引导区**  
   放一句明确定位文案、一个搜索框，以及三张按学习目标切入的路线卡：
   - 系统入门 AI Agent
   - 补齐工程化能力
   - 边学边做项目

2. **推荐起点区**  
   当用户选择某个目标后，在当前页内展开推荐起点，直接告诉用户：
   - 先读什么
   - 先做什么
   - 读完或做完下一步去哪里

3. **按内容类型浏览区**  
   为想自己逛的人提供类型化入口：
   - 章节学习
   - 实践项目
   - 学习路线
   - 进阶专题

4. **主题聚合与延伸发现区**  
   以主题为索引组织关键内容，例如工具调用、会话状态、规划机制、记忆、RAG 与上下文工程，并让每个主题同时出现基础章节、进阶内容与实践项目。

整体交互顺序应遵循：
- 先选目标
- 再看推荐起点
- 再按类型或主题继续逛

这样 `/discover` 更像学习分诊台，而不是内容仓库目录。

### 3. 搜索保留现有引擎，但必须增强“识别性”和“选择性”

Phase 5 不建议替换 VitePress 本地搜索，而是在其之上补一层课程化识别能力。搜索增强的关键不是“搜得更多”，而是“搜到后看得懂、选得准”。

搜索结果至少要补足三类信息：

1. **内容类型标识**  
   每条结果都应清楚标明它是章节、实践项目、学习路线还是进阶专题。

2. **学习路径语义**  
   结果要能表达它处于哪条学习链路，例如入门基础、实践跟练、进阶专题，而不是只有页面标题。

3. **简短的选择理由**  
   告诉用户这条结果更适合哪类场景，例如“适合第一次建立概念”“适合补工具调用工程化”“适合直接跟练项目”。

搜索增强的目标是让本地搜索从“文档检索入口”升级为“学习内容选择器”，但不把 Phase 5 变成高风险搜索引擎改造。

### 4. 内容类型要从内部 metadata 升级为全站一致的前台语言

当前已有的 `contentType` 和相关 metadata 主要服务内部建模，Phase 5 需要把它们转化为用户稳定可感知的类型系统。

建议统一为四类前台标签：

- `章节`
- `实践项目`
- `学习路线`
- `进阶专题`

这套标签必须在三个场景中保持一致：

1. `/discover` 的推荐卡  
2. 搜索结果  
3. 内容页中的相关推荐与下一步入口  

当用户在站内反复看到同一套类型语言时，会更快建立心智模型，不再把所有页面都看成“普通文章”。

### 5. 推荐关系要从“相关内容”升级为“学习关系”

Phase 3 和 Phase 4 已经形成了一部分推荐数据，Phase 5 需要把这些关系抽象为更明确的学习关系，而不是停留在“你可能还想看”。

至少需要统一以下三类关系：

1. **下一步推荐**  
   解决“看完这一页后应该去哪里”。

2. **相关实践**  
   解决“这个知识点能拿去做什么”，尤其适用于章节页。

3. **前置补课**  
   解决“如果我看不懂或做不动，该先补什么”，尤其适用于实践项目页和进阶专题页。

这样用户在任意入口进入后，都能沿着“前置 -> 当前 -> 下一步”的链路继续前进，而不是回到目录重新搜索。

### 6. Phase 5 应新增 discovery 编排层，而不是再造一套数据体系

现有真源已经足够丰富：

- `content-meta.ts`
- `content-index.data.ts`
- `learning-paths.data.ts`
- `practice-projects.ts`
- 各页面已有的 `contentType`、`searchTags`、`navigationLabel`、`entryMode` 等字段

因此本阶段最稳的做法不是重新维护一份大型内容配置，而是新增一层 discovery view model，专门负责：

1. 按学习目标组织推荐起点  
2. 按主题聚合章节、实践和专题  
3. 向前台输出统一的内容类型和推荐文案  
4. 串联前置、下一步和相关实践等学习关系  

这个编排层应成为 `/discover`、搜索增强组件和局部相关推荐区的共享真源，避免把推荐逻辑散写在多个页面和组件中。

### 7. 展示层应保持可降级，避免发现体验成为新的脆弱点

由于当前仍是静态文档站，Phase 5 的增强必须遵循“缺一块数据也不影响页面可用”的原则：

1. 推荐数据缺失时，页面应回退到基础内容卡，而不是整块崩掉。  
2. 内容类型字段不完整时，应有默认类型展示，而不是出现空标签。  
3. 搜索增强信息缺失时，至少保留原始标题跳转能力。  
4. 推荐关系不完整时，应回退到同主题或同类型的基础推荐。  

这样即使后续新增内容时元数据还不完整，站点也不会出现明显断层。

## Phase 5 拆分

### 05-01 搜索索引与结果识别增强

目标：
- 保留本地搜索引擎
- 增强搜索结果的内容类型、路径语义和选择提示
- 为搜索和发现共享统一的前台类型语言

### 05-02 统一发现中心与推荐关系收口

目标：
- 新增 `/discover` 作为统一学习起点
- 建立按目标、按主题、按类型的发现结构
- 收口首页引流入口和关键页的推荐关系

---

## 风险与约束

1. Phase 5 不做登录、云端同步、真实进度系统、AI 助教、评论区、题库评分和视频化内容。
2. 不替换现有搜索引擎，只增强索引字段、结果识别和结果周边体验。
3. 不把 `/discover` 做成新的信息堆叠页，首屏必须优先服务“第一次选择”。
4. 不引入第二套内容元数据体系，发现能力必须建立在现有真源和新增编排层之上。
5. 所有推荐增强都应支持降级，不能因为个别内容缺字段就破坏整页体验。

## 验证方式

自动验证：
- `bun run build:strict`
- 新增或扩展发现体验校验脚本，至少覆盖 `/discover` 关键区块、内容类型字段和推荐关系最小完整性

人工验证：
- 从首页进入站点，确认可以明显找到“发现 / 开始学习”入口并跳转到 `/discover`
- 打开 `/discover`，确认首屏以“按目标选路线”为主，而不是一堆平铺链接
- 在 `/discover` 中任选一个目标路线，确认页面能给出明确的先读、先做和下一步建议
- 使用站内搜索搜索如“工具”“规划”“记忆”等主题，确认结果可以区分内容类型
- 打开任一关键章节和任一实践项目页，确认至少能看到下一步或相关实践/前置补课入口
- 检查移动端下 `/discover` 首屏、路线卡和内容卡的层级与折行是否稳定
