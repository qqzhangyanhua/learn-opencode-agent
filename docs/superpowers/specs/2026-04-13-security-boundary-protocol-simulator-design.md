# 第31章安全协议状态机设计稿

## 背景

第 31 章当前已经有 `SecurityBoundaryDemo`，但它更像一个“规则命中检查器”：

- 用户切换场景
- 点击执行安全检查
- 规则层给出命中结果
- 最后展示允许或拦截

这个演示能说明“安全规则存在”，但还没有完整呈现第 31 章真正强调的主链：

- 先做风险分级
- 再给最小权限基线
- 高风险动作进入确认机制
- 工具执行前再做运行时校验

也就是说，现在读者能看到“拦截结果”，却不容易建立“为什么系统不是在最后一步才突然拦截”的心智模型。

## 目标

把现有 `SecurityBoundaryDemo` 升级成“安全协议状态机”，让读者沿着 `风险分级 -> 最小权限 -> 确认机制 -> 运行时校验` 看到一次请求是怎样被层层收口的。

成功标准：

- 组件顶部有四阶段切换
- 当前场景仍可切换
- 每个阶段都展示不同的判断重点
- 对高风险场景能清楚看到“为什么需要人工确认”
- 最终 verdict 不再像黑盒结果，而是可追溯到前面每一层判断

## 范围

本次只升级第 31 章及现有 `SecurityBoundaryDemo`。

包含：

- 升级 `SecurityBoundaryDemo.vue`
- 如有必要，给 `types.ts` 补最小辅助类型
- 在章节体验校验里增加第 31 章状态机要求

不包含：

- 不新建平行的大型安全组件
- 不引入真实权限服务端逻辑
- 不把第 31 章正文大改

## 方案选择

### 方案 A：升级现有 `SecurityBoundaryDemo`

做法：

- 保留当前场景与规则数据
- 在组件内部增加四阶段状态
- 让右侧结果区升级为“协议链路展示”

优点：

- 改动范围可控
- 与当前章节内容衔接最自然
- 不会出现两个安全演示组件职责重叠

缺点：

- 需要重构现有组件布局和状态

### 方案 B：新增 `SecurityProtocolSimulator`

做法：

- 新建章节级组件
- 保留旧的 `SecurityBoundaryDemo`

优点：

- 新旧职责完全分离

缺点：

- 第 31 章组件入口会重复
- 对当前章节收益不如直接升级现有组件

### 推荐方案

采用方案 A。

原因：

- 第 31 章不是缺少“另一个视觉容器”，而是缺少“安全边界是一条协议链”的显式状态切换
- 在现有组件上升级即可完成教学目标，成本更低

## 交互结构

### 顶部阶段条

四个阶段：

- 风险分级
- 最小权限
- 确认机制
- 运行时校验

切换阶段时：

- 不改变当前选中的场景
- 不改变规则集
- 只改变主视图区与右侧解释

### 场景切换

继续保留现有场景切换，例如：

- Prompt 注入
- 合法请求

必要时可增加第三个中间态场景：

- 高风险但可确认放行

这样更能体现“不是只有 allow/block 两种世界”。

## 四阶段设计

### 阶段一：风险分级

目标：

- 先判断这次请求的后果等级，而不是先讨论工具名

展示重点：

- 当前请求涉及哪些风险点
- 风险等级如何决定后续处理强度

解释重点：

- 风险不是按“是不是工具调用”分
- 而是按做错后的破坏半径分

### 阶段二：最小权限

目标：

- 展示当前角色默认能做什么、不能做什么

展示重点：

- 角色默认权限基线
- 哪些目录 / 能力可访问
- 哪些能力即使模型想用也默认不可见

解释重点：

- 不是先给满权限，再期待模型自觉克制
- 而是先收紧，再按规则升级

### 阶段三：确认机制

目标：

- 展示高风险动作为什么必须挂起等待批准

展示重点：

- 是否触发人工确认
- 当前动作是等待批准、被拒绝还是放行
- 为什么确认不是一个 UI 提示，而是执行协议

解释重点：

- 未批准前不能执行
- 确认必须和执行循环挂起绑定

### 阶段四：运行时校验

目标：

- 展示最终工具调用前的最后一道硬边界

展示重点：

- 路径是否越界
- 工具是否可见
- 参数是否越权
- 最终 verdict：allow / block / require approval / degrade

解释重点：

- Prompt 只能影响倾向
- 真正的兜底要在代码边界

## 视觉结构

保留当前“三栏感”，但改成：

- 左侧：当前输入 / 场景
- 中部：当前阶段的安全协议画布
- 右侧：最终状态 / 命中规则 / 当前阶段说明

底部或顶部补一条阶段切换按钮组。

## 数据设计

尽量复用当前：

- `rules`
- `scenarios`

只补最小派生数据，不新增复杂 schema。

可在组件内部推导：

- 当前场景风险等级
- 当前角色权限基线
- 是否需要确认
- 运行时 verdict

如果必须补类型，只在 `types.ts` 里增加：

- `SecurityStageKey`
- `SecurityPermissionBaseline`

## 章节接入方式

第 31 章正文仍保留 `<SecurityBoundaryDemo />`，不改标签名。

正文只补一句操作提示：

“先切换四个安全阶段，再观察同一个请求是怎样被逐层收口的。”

- `bun run build`

章节体验校验新增要求：

- `SecurityBoundaryDemo.vue` 必须包含阶段切换处理函数
- 必须包含四阶段标识：
  - 风险分级
  - 最小权限
  - 确认机制
  - 运行时校验

## 风险与控制

### 风险 1：把安全协议演成纯视觉流程

控制：

- 每阶段都要有明确判断结果
- 不只做动画，不空转状态

### 风险 2：高风险场景只有 block，没有“确认后可执行”中间态

控制：

- 至少补一个“需要确认”的中间场景，体现安全协议不是简单二元判断

### 风险 3：和正文术语脱节

控制：

- 所有阶段名称直接取自正文主链
- 说明文案围绕“风险、权限、确认、校验”四个关键词
