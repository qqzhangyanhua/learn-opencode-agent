# 第08章 HTTP API 权限守门对照组件设计稿

## 背景

第 08 章当前虽然已经把 `createApp` 的中间件链、BasicAuth、CORS、上下文注入和 SSE 讲清楚了，但读者在阅读时最容易被这些概念并列冲散注意力。

这一章真正应该优先记住的，不是所有中间件细节，而是：

```text
HTTP 请求
  -> 权限中间件
  -> 路由匹配
  -> Handler
  -> 响应返回
```

并且用户已经明确希望本章首先突出：

- 主链优先
- 中间件是主角
- 在中间件里优先强调“权限校验决定请求能做到哪一步”
- 用“同一请求，两种结果对照”的方式加深印象

## 目标

给第 08 章新增一个章节级教学组件，用同一个请求的双栏对照，让读者直接看出：

- 请求不是直接进入 Handler
- 权限中间件才是入口守门者
- 没权限的请求会停在中间件并直接返回
- 有权限的请求才会继续进入路由和业务处理

成功标准：

- 顶部固定显示 `HTTP 请求 -> 权限中间件 -> 路由匹配 -> Handler -> 响应返回`
- 中间主舞台以“同一请求的双结果对照”为主
- 右侧固定说明四个记忆问题
- 自动播放时先走到权限中间件，再分叉成拒绝和放行两条路径
- 不做炫技动画，只做轻量高亮、推进和分叉

## 方案选择

### 方案 A：单链路推进

只做一条请求主链，按阶段高亮推进，最后根据权限结果停下或继续。

优点：

- 改动小
- 易于实现

缺点：

- “有权限 / 无权限”对照不够强
- 用户不容易一眼记住权限校验的分叉价值

### 方案 B：同一请求双栏对照

同一个 `POST /session/:id/message` 请求，左边走“权限不足 -> 403”，右边走“权限通过 -> 路由 -> Handler -> 200 / stream”。

优点：

- 最符合本章当前教学目标
- 用户能直接把“权限决定请求能走多远”记住
- 结构上也更适合复用最近几章的“主链 + 对照 + 记忆面板”模式

缺点：

- 需要维护两条路径的状态高亮

### 方案 C：多关卡切页

把身份、权限、参数校验拆成多个关卡，让用户逐个切换。

优点：

- 表达完整

缺点：

- 会把本章重心带偏到“安全专题”
- 与用户确认的“权限优先 + 双结果对照”不一致

### 推荐方案

采用方案 B。

## 交互结构

### 顶部主链

固定五段：

- HTTP 请求
- 权限中间件
- 路由匹配
- Handler
- 响应返回

要求：

- 可点击切换当前说明
- 自动播放时随流程推进高亮

### 中间主舞台

双栏并排：

- 左栏：权限不足路径
- 右栏：权限通过路径

两栏共用同一个请求文案，例如：

`POST /session/:id/message`

左栏流程：

- 请求进入
- 命中权限中间件
- 校验失败
- 返回 `403 Forbidden`
- 路由和 Handler 不再执行

右栏流程：

- 请求进入
- 命中权限中间件
- 校验通过
- 进入路由匹配
- Handler 执行业务
- 返回正常响应

### 右侧固定说明

至少包含四个记忆字段：

- 谁在做守门
- 拦住的依据是什么
- 被拦住后请求停在哪里
- 放行后为什么 Handler 可以更简单

再补两个辅助区域：

- 当前阶段说明
- 一句话记忆

## 文案原则

一句话记忆固定为：

`API 请求能走多远，首先不是看 Handler 会不会处理，而是看入口权限中间件放不放行。`

右侧说明必须围绕“权限守门”，不要扩写成日志、SSE、上下文注入总览。
