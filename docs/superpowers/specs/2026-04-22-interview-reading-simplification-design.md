# 面试专区阅读减负设计稿

## 背景

当前面试专区每道题采用固定结构：

- `题目`
- `面试官想听什么`
- `短答版`
- `深答版`
- `追问方向`

问题不在于内容不够清楚，而在于这些信息默认同时展开，导致单题阅读时视觉重心不稳定。用户在看“题目”和“短答版”时，会被同层级出现的长段 `深答版` 打断，产生眼花和阅读疲劳。

这个问题是真实问题，且不需要重做布局。

## 目标

在不重构页面样式、不引入新组件、不改文案结构的前提下，降低单题阅读负担。

成功标准：

- 用户进入单题后，默认阅读路径变成 `题目 -> 短答版 -> 追问方向`
- `深答版` 改为按需展开
- 原有 Markdown 内容结构仍然容易维护
- 不改变导航、信息架构和页面气质

## 核心判断

这是一个值得做的小优化，但只值得做最小改动。

不值得做的方向：

- 重做卡片化布局
- 主次分栏
- 自定义 Vue 组件包装每道题
- 增加过强的视觉装饰和交互动画

原因：

- 当前问题是“默认暴露太多”，不是“页面没有设计”
- 用更重的结构解决轻问题，会把内容页做成产品界面，反而破坏阅读稳定性
- Markdown 内容未来还会持续扩充，过重结构会提高维护成本

## 方案选择

### 方案 A：只折叠 `深答版`

保留现有题目结构，只把每道题的 `深答版` 包进原生折叠容器。

优点：

- 改动最小
- 阅读收益直接
- 兼容当前内容写法
- 风险最低

缺点：

- 页面整体视觉不会发生明显变化

### 方案 B：折叠 `深答版`，并补少量样式

在方案 A 的基础上，为折叠标题、间距和边框补少量统一样式。

优点：

- 阅读层级更清楚
- 仍然是低风险改动

缺点：

- 会多出一层样式维护
- 如果样式过重，容易再次走向过度设计

### 方案 C：自定义题目组件

把面试题抽成统一组件，内部处理层级、折叠和样式。

优点：

- 结构最统一

缺点：

- 明显超出当前问题规模
- 提高内容维护成本
- 不符合最小必要原则

### 推荐方案

采用方案 A，必要时最多加方案 B 中非常轻的一点样式。

## 设计细则

### 默认可见内容

每道题默认直接展示：

- `题目`
- `面试官想听什么`
- `短答版`
- `追问方向`

### 默认折叠内容

每道题的 `深答版` 默认折叠，用户需要时再展开。

折叠标题统一为：

- `展开深答版`

展开后展示完整原文，不做摘要重写，不拆段重排。

### 内容顺序

保留现有题目内部顺序，不重排字段位置。

原因：

- 当前问题主要来自信息默认展开，而不是字段顺序失效
- 保持最小 diff，避免大面积改 Markdown 内容

### 技术实现方向

优先使用站点已存在的 Markdown 折叠能力：

- `::: details`
- 或原生 `<details><summary>`

具体选型以当前仓库中更稳定、书写更一致的方式为准。

### 样式边界

如果需要补样式，只允许做这些轻量增强：

- 折叠区上下留白更清楚
- summary 标题字重略高
- 展开内容与正文之间保持弱边界

禁止做这些增强：

- 大面积卡片背景
- 左右分栏
- 题目块重新组件化
- 明显动画和复杂交互反馈

## 范围

本次设计包含：

- 面试专区各分类页的单题阅读优化
- `深答版` 折叠策略
- 必要的极轻量样式支持

本次设计不包含：

- 面试专区导航改造
- 题目筛选
- 题目分页
- 题库模式 / 模拟面试模式切换
- 新组件抽象

## 风险与兼容性

主要风险：

- 如果折叠标题文案不统一，会造成页面观感不整齐
- 如果不同页面手工改法不一致，后续维护会出现格式漂移

兼容性要求：

- 桌面端和移动端行为一致
- 不影响现有链接、标题锚点和章节结构
- 不破坏当前文档的纯 Markdown 可编辑性

## 验收标准

实现完成后应满足：

- 所有面试分类页中的 `深答版` 默认不再直接展开
- 用户可以显式点击 `展开深答版` 查看详细内容
- 单题阅读时，首屏首先看到的是题目和短答版，而不是长篇说明
- 页面整体仍保持现有站点的文档页风格
