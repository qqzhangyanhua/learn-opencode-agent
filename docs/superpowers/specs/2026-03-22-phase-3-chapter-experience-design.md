# Phase 3 章节学习体验标准化设计文档

**日期**: 2026-03-22  
**范围**: 核心章节页顶部学习摘要、底部下一步与练习闭环、章节间推荐关系  
**依赖**: Phase 1 的 `content-meta.ts` / `content-index.data.ts`，Phase 2 的入口页与导航改造

---

## 背景

Phase 1 解决了“内容怎么分类、怎么建模”，Phase 2 解决了“用户从哪里进入”。但用户一旦真正点进章节，站点仍然有明显的文档阅读惯性：

1. **章节页缺少统一的学习入口信息**  
   虽然 frontmatter 已经包含 `learningGoals`、`prerequisites`、`recommendedNext`、`practiceLinks`、`estimatedTime` 等字段，但这些信息还没有被统一渲染成稳定的学习界面。

2. **理论 / 实践 / 中级三类章节体验不一致**  
   理论篇大多是手写 blockquote，实践篇部分页面有 `ProjectCard`，中级篇又是另一种写法。用户点进不同板块时，缺少统一的“我为什么读、读前要准备什么、读完下一步做什么”体验。

3. **章节结尾缺少强约束的任务闭环**  
   当前部分章节有推荐阅读，但不稳定，也缺少明确练习动作或对应实践入口，难以满足 `CHAP-03` 和 `CHAP-04`。

Phase 3 的目标不是重写章节正文，而是在不破坏现有内容主体的前提下，把“章节页”升级成真正的学习单元。

---

## 成功标准

这一阶段完成后，下面四件事必须为真：

1. 用户打开任一首批标准化章节时，首屏就能看到学习目标、适合人群、预计投入时间和前置要求。
2. 用户在章节内不需要回忆路径页，也能知道该章属于哪条学习链路、适合在哪个阶段阅读。
3. 用户读到章节结尾时，能看到明确的下一步推荐，并至少有一个可执行练习、任务或对应实践入口。
4. 理论篇、实践篇和中级篇的首批章节虽然内容不同，但学习结构一致，不再各写各的。

---

## 方案比较

### 方案 A：每章手写提示块

继续沿用 markdown 手写 blockquote、列表和 CTA，在每章里单独补齐学习目标、前置知识和下一步。

优点：
- 实现快
- 不需要新增组件

缺点：
- 极易再次出现样式和结构漂移
- 后续扩展到更多章节时维护成本高
- 无法真正复用 frontmatter 数据

### 方案 B：frontmatter 驱动的章节学习组件

新增少量章节组件，统一读取当前页 frontmatter 和内容索引，把“学习摘要 + 下一步 / 练习”渲染为固定结构；仅在少数页面补充轻量 props 作为动作闭环。

优点：
- 最大化复用 Phase 1 既有字段
- 组件边界清晰，适合逐步扩展到更多章节
- 能同时覆盖理论 / 实践 / 中级三类页面

缺点：
- 需要设计一套兼容三类页面的通用组件
- 仍需要给首批章节补少量动作内容

### 方案 C：直接重写章节布局

优先改造 VitePress 章节布局，在全局层面强制章节页套入新的 hero、目录和底部结构。

优点：
- 全站统一程度最高

缺点：
- 风险过高，容易打断现有正文与目录体验
- 当前阶段不需要全局布局级重构

**推荐方案：B。**

理由：Phase 3 的本质是“把已有章节 metadata 真正用起来”，而不是再开启一轮大规模结构重做。方案 B 能在最小改动下，把章节页变成标准化学习单元，同时为 Phase 4 的实践页课程化重构提供模板。

## 设计结论

### 1. 章节页要从“文章”升级成“学习单元”

首批标准化章节需要稳定呈现两层结构：

1. **页首学习摘要**  
   告诉用户这章是什么、为什么现在读、需要准备什么、预计投入多久。

2. **页尾行动闭环**  
   告诉用户读完后下一步去哪里、先做什么练习、对应哪个实践或专题入口。

正文主内容保持不动，避免 Phase 3 变成“重写整章”。

### 2. 优先复用 frontmatter，而不是重新发明章节配置

现有 frontmatter 已覆盖：

- `learningGoals`
- `prerequisites`
- `estimatedTime`
- `recommendedNext`
- `practiceLinks`
- `entryMode`
- `roleDescription`

Phase 3 不应重开全站 metadata 合同。组件优先消费这些字段；只有“本章练习 / 行动任务”这类当前缺失的信息，才在页面内用轻量 props 补充。

### 3. 首批只标准化 7 个高价值章节

为了避免一次铺开全书导致范围膨胀，Phase 3 只处理这 7 个关键章节：

- `docs/00-what-is-ai-agent/index.md`
- `docs/01-agent-basics/index.md`
- `docs/03-tool-system/index.md`
- `docs/04-session-management/index.md`
- `docs/practice/p01-minimal-agent/index.md`
- `docs/practice/p10-react-loop/index.md`
- `docs/intermediate/27-planning-mechanism/index.md`

它们分别承接：

- 首页与学习路径的核心起点
- 理论主链路的关键节点
- 实践篇的第一跳与升级跳
- 中级篇的桥接入口

### 4. 统一组件应拆成“顶部摘要 + 底部行动”两个职责

推荐新增两类组件：

- `ChapterLearningGuide.vue`  
  顶部学习摘要组件，负责显示：
  - 当前章节定位
  - 学习目标
  - 适合人群 / 阅读阶段
  - 预计投入时间
  - 前置知识

- `ChapterActionPanel.vue`  
  底部行动闭环组件，负责显示：
  - 推荐下一步
  - 对应实践 / 理论 / 中级回链
  - 一条可执行练习或任务

其中顶部组件尽量零配置读取 frontmatter，底部组件允许通过 props 补充“本章行动任务”。

## Phase 3 拆分

### 03-01 统一章节模板与组件

目标：
- 建立章节学习摘要和行动闭环的复用组件
- 把 metadata 从 frontmatter 变成真正可见的学习界面

### 03-02 首批关键章节接入

目标：
- 为 7 个关键章节补齐一致的学习结构
- 让理论 / 实践 / 中级篇首次出现统一章节体验

### 03-03 章节推荐关系与动作闭环

目标：
- 把 `recommendedNext` / `practiceLinks` 变成真正的跨章节推荐卡
- 为首批章节补齐至少一个可执行任务或对应实践入口

---

## 风险与约束

1. Phase 3 不做全站章节布局重写，只在现有页面中插入统一组件。
2. 不重新修改全量 metadata 合同，避免牵动所有章节和校验脚本。
3. 首批只标准化 7 个高价值章节，不提前把全书全部模板化。
4. 练习与行动任务以轻量、可执行为原则，不做题库、判题或进度系统。

---

## 验证方式

- `bun run build:strict`

- 打开首批 7 个章节，确认顶部有统一学习摘要结构
- 打开首批 7 个章节，确认底部有下一步推荐和可执行行动
- 检查理论 / 实践 / 中级三类页面的组件视觉和信息层次是否一致
- 检查移动端下顶部摘要卡和底部行动卡是否折行正常
