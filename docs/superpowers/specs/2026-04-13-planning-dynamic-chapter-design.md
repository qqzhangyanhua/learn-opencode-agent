# Planning 动态章节设计说明

**日期：** 2026-04-13  
**状态：** 已确认设计，待进入实现规划  
**适用范围：** 电子书章节模板试点  
**试点目标：** 将 Planning / 任务分解主题升级为动态流程型章节，提升理解效率、记忆强度与后续复刻意愿

## 1. 设计目标

这个试点章节不是普通长文，也不是纯动画演示，而是一个以用户选择驱动的 `Planning 流程模拟器`。

目标有三个：

1. 让用户通过亲手决策，看懂简化版 Planning 机制如何运转
2. 让用户清晰记住 Planning 为什么必要、何时需要重规划
3. 让用户在体验结束后，具备复刻简化版 Planning 机制的结构直觉

## 2. 章节定位

这是一个 `B 类动态流程型章节`。

相对普通解释型章节，它强调：

- 过程可见
- 状态变化可见
- 错误路径可见
- 重规划过程可见

相对理论-实践桥接型章节，它当前优先级更偏 `交互理解`，而不是直接讲实现代码。

## 3. 用户体验原则

本章节遵循以下原则：

1. `动态负责让用户看见过程`
2. `文字负责让用户记住原理`
3. `每一步只做一个关键决策`
4. `每次决策都必须有即时反馈`
5. `先体验，再命名，再抽象`

为了避免“好看但没学会”，内容比例控制为：

- 60% 动态与交互
- 25% 短解释文字
- 15% 总结与迁移提示

## 4. 页面骨架

章节采用 6 屏结构。

### 第 1 屏：任务挑战页

作用：

- 把用户拉进一个真实复杂任务
- 建立“为什么需要 Planning”的问题场景

页面元素：

- 任务目标卡片
- 一句挑战说明
- 开始按钮
- 简短提示：这是一段交互式流程模拟，而不是普通阅读页

### 第 2 屏：是否先规划

作用：

- 打出第一层核心冲突
- 让用户亲手体验“直接执行”和“先规划”的差异

用户选择：

- 直接执行
- 先做高层规划

预期反馈：

- 直接执行时暴露目标混杂、执行风险升高
- 先规划时进入任务树生成流程

### 第 3 屏：任务树拆解页

作用：

- 让用户理解任务拆分粒度是 Planning 的核心决策之一

用户选择：

- 拆得很粗
- 拆得适中
- 拆得很细

预期反馈：

- 中央任务树即时重绘
- 右侧解释每种粒度的代价与风险
- 明确展示：拆得过粗与过细都不是最优

### 第 4 屏：执行推进页

作用：

- 让用户看到 Planning 与执行并不是两张皮

用户选择：

- 先执行哪个子任务
- 是否并行推进
- 是否继续当前路径

预期反馈：

- 任务节点状态变化
- 依赖关系暴露
- 顺序选择的合理性被解释清楚

### 第 5 屏：失败与重规划页

作用：

- 讲透 Planning 的真正价值不只是拆任务，而是动态修正路径

系统事件：

- 信息不足
- 当前路径阻塞
- 子任务划分不合理

用户选择：

- 硬做下去
- 局部修补
- 回到上层重规划

预期反馈：

- 旧任务树收缩
- 新任务树生成
- 明确说明重规划不是推翻重来，而是基于当前结果重新组织任务

### 第 6 屏：总结与复刻页

作用：

- 把体验收束为结构化认知
- 帮用户形成可迁移的复刻直觉

页面元素：

- 用户本轮路径回放
- 关键决策总结
- 简化版 Planning 机制最小模块图
- 推荐跳转：相关理论章、实践项目、源码位置

## 5. 核心交互结构

模拟器采用四栏协同结构。

### 顶部：当前任务

固定显示当前目标，避免用户在多步交互中丢失上下文。

### 左侧：决策面板

职责：

- 提供当前步骤唯一关键决策
- 严格限制为 2 至 3 个可选项

要求：

- 不允许同屏出现过多控制项
- 选项文案必须短、明确、可对比

### 中间：任务树与状态变化

职责：

- 作为核心视觉记忆区
- 承担任务树展开、节点状态变化、重规划前后对比

要求：

- 节点状态必须明显区分
- 路径切换必须可追踪
- 重规划前后必须形成清晰对照

### 右侧：系统反馈

职责：

- 解释刚才发生了什么
- 命名当前阶段的抽象概念
- 说明当前选择为什么更优或更差

要求：

- 每段文字保持短解释，不写成长文
- 必须服务于刚刚发生的状态变化

### 底部：阶段条

建议阶段：

1. 接收目标
2. 初步规划
3. 子任务拆解
4. 执行推进
5. 结果检查
6. 重规划

作用：

- 帮助用户始终知道自己处于哪一段流程
- 增强阶段性记忆

## 6. 必须动态化的内容

以下内容必须做成动态表达：

1. 任务树生成
2. 节点状态变化
3. 错误路径暴露
4. 重规划前后对比
5. 用户选择后的即时反馈

如果这些部分仅用文字讲述，本章节将失去试点价值。

## 7. 必须保留为文字的内容

以下内容必须保留文字表达：

1. 这一步为什么重要
2. 关键术语命名
3. 最终抽象总结
4. 简化版复刻提示

文字不应承担主流程解释，但必须承担“命名”和“收束”的职责。

## 8. 视觉语言

推荐使用以下混合视觉语言：

- 主基调：`系统流程感`
- 局部骨架：吸收 `终端工程感`
- 记忆高点：少量使用 `课程实验室感`

### 视觉目标

1. 清晰稳定，不做无意义炫技
2. 具有工程质感，符合 Agent / Planning 主题语境
3. 在重规划成功、总结收束等时刻形成明显记忆高点

### 不推荐方向

- 全章强动画化：容易形成观看疲劳
- 纯终端硬核感：容易重新变成阅读疲劳
- 纯课程乐园感：会削弱工程可信度

## 9. 信息架构

为保证工程理性与教学感染力平衡，建议每屏都遵守如下结构：

1. 一个目标
2. 一个选择
3. 一个变化
4. 一个解释
5. 一个阶段定位

即：

`目标 -> 决策 -> 系统变化 -> 原理解释 -> 阶段归位`

## 10. 组件建议

实现阶段建议拆分为下列组件或模块：

- `PlanningMissionCard`：任务挑战卡
- `PlanningDecisionPanel`：左侧决策面板
- `PlanningTreeCanvas`：中部任务树与状态变化区
- `PlanningFeedbackPanel`：右侧系统解释
- `PlanningStageBar`：底部阶段条
- `PlanningReplaySummary`：结尾回放与总结

当前文档只定义职责，不约束具体技术实现细节。

## 11. 状态模型建议

任务节点建议至少包含以下状态：

- `idle`
- `planned`
- `running`
- `blocked`
- `failed`
- `done`
- `replanned`

规划阶段建议至少包含以下枚举：

- `goal`
- `outline`
- `decompose`
- `execute`
- `review`
- `replan`

这些状态命名既服务交互实现，也服务后续教学术语统一。

## 12. 错误处理与边界

为了避免模拟器变成混乱分支树，必须控制边界：

1. 每屏只出现一个关键决策
2. 每个决策只提供 2 到 3 个选项
3. 分支不做无限延展，而是回收到固定学习目标
4. 所有错误路径最终都应导向“为什么需要重规划”的教学结论

## 13. 验收标准

这个试点章完成后，至少应满足以下标准：

1. 用户可以独立走完一轮 Planning 生命周期
2. 用户能明确感知“先规划”和“直接执行”的差异
3. 用户能看懂任务树如何因选择而变化
4. 用户能理解重规划的触发原因与价值
5. 用户在结尾能说出简化版 Planning 机制的最小结构

## 14. 测试与验证建议

实现后建议做三类验证：

### 内容验证

- 文字是否压缩到必要程度
- 动态是否真正承载理解，而不是装饰

### 交互验证

- 每屏决策是否足够清楚
- 每次决策后系统变化是否一眼可见

### 学习验证

- 用户是否能在结束后复述最小 Planning 流程
- 用户是否知道去哪一章、哪个实践项目继续

## 15. 后续扩展方向

如果试点成功，可以沿三条线扩展：

1. 将 Tool Call 生命周期升级为同类动态流程章
2. 将 Session / 压缩过程升级为同类动态流程章
3. 在第二版中增加代码映射侧栏，逐步升级为理论-实践桥接模板

## 16. 结论

本试点章的本质不是“给一章加动画”，而是建立一种新的章节生产方式：

- 普通文字章节负责解释
- 动态流程章节负责让用户看见机制
- 后续桥接章节负责把理解迁移到代码与实践

对于 Planning 主题，最正确的切入不是定义堆叠，而是让用户通过做选择，看到一个简化版 Planning 机制如何生成、执行、失败、再重规划。

---

**当前结论：**

- 采用 `B 类动态流程型章节`
- 采用 `流程模拟器式` 结构
- 优先满足 `交互理解`
- 风格定位为 `工程理性 + 教学感染力` 平衡
- 作为后续动态章节模板的首个试点
