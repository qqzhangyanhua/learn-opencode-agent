# 第32章性能与成本四阶段仪表盘设计稿

## 背景

第 32 章当前的 `CostOptimizationDashboard` 已经能展示：

- 基线成本
- 优化后成本
- 各项 token / cost 明细
- 优化建议

但它更像一个“结果型成本看板”，还没有完整呈现第 32 章真正强调的主链：

- 先识别任务复杂度
- 再选择合适模型
- 同时给上下文和工具输出设预算
- 最后把成本与延迟变成可观察指标

也就是说，现在读者能看到“优化前后省了多少钱”，却不容易建立“为什么这些钱是这样省下来的”的工程心智模型。

## 目标

升级现有 `CostOptimizationDashboard`，让它变成四阶段状态面板，围绕：

- 复杂度识别
- 模型路由
- 预算控制
- 可观测性

把“性能与成本是架构约束”这条主链讲清楚。

成功标准：

- 顶部有四阶段切换
- 每一阶段都有独立的解释重点
- 现有“基线 / 优化后”成本看板保留，但变成某一阶段的子视图，而不是整个组件的全部
- 右侧始终显示当前阶段关注点、风险和工程动作
- 组件仍保留 `CostOptimizationDashboard` 名称，不改正文标签

## 范围

本次只改第 32 章和现有 `CostOptimizationDashboard.vue`。

包含：

- 升级 `CostOptimizationDashboard.vue`
- 在正文中补一句操作提示
- 在章节体验校验中增加第 32 章四阶段回归要求

不包含：

- 不新建平行的大型性能成本组件
- 不接入真实 tracing、metrics 或计费后端
- 不重写整章正文

## 方案选择

### 方案 A：升级现有 `CostOptimizationDashboard`

做法：

- 保留现有组件名和成本数据
- 在组件内部增加四阶段状态
- 将现有“基线/优化后对比”嵌入预算控制阶段

优点：

- 接入成本最低
- 复用已有数据和视觉资产
- 章节主线能直接在一个组件里闭环

缺点：

- 组件内部状态比原来更复杂

### 方案 B：新增 `PerformanceCostControlPanel`

做法：

- 新建章节级组件
- 保留旧组件

优点：

- 新旧职责分离

缺点：

- 对当前章节会形成重复入口
- 成本大于收益

### 推荐方案

采用方案 A。

原因：

- 当前缺的不是“更多图表”，而是“成本优化的决策链路”
- 直接升级现有组件即可达到教学目标

## 四阶段设计

### 阶段一：复杂度识别

展示重点：

- 当前请求属于简单 / 中等 / 复杂哪一层
- 为什么复杂度判断先于模型选择

目标：

- 让读者理解不是所有请求都值得上最强模型

### 阶段二：模型路由

展示重点：

- 不同复杂度映射到不同模型层级
- 预算紧张时如何进一步降级

目标：

- 让模型选择从感性动作变成显式策略

### 阶段三：预算控制

展示重点：

- 继续复用当前“基线 / 优化后”成本条形图
- system prompt、历史、输出、工具结果分别怎么被压缩或缓存

目标：

- 让用户看见“无效 token”才是成本大头

### 阶段四：可观测性

展示重点：

- input / output token
- latency
- retry
- tool truncation
- model route

目标：

- 让用户理解“不可见的成本无法被管理”

## 视觉结构

建议结构：

- 顶部：标题 + 四阶段切换
- 左侧 / 中部：当前阶段主视图
- 右侧：阶段说明、风险提示、工程动作

保留当前卡片和条形图风格，不追求炫技动画。

## 数据设计

继续使用现有 `scenarios` prop。

组件内补最小本地数据：

- 复杂度样例
- 模型路由策略
- 可观测性指标样例

原则：

- 不修改全站 `CostScenario` schema
- 不新增复杂外部数据源

## 章节接入方式

正文继续使用 `<CostOptimizationDashboard />`。

只在组件前补一句：

“按顺序切换四个阶段，观察性能和成本是在哪一层开始失控的。”

## 验证方案

至少验证：

- `bun run typecheck`
- `bun run check:chapter-experience`
- `bun run build`

章节体验校验新增要求：

- `CostOptimizationDashboard.vue` 必须包含阶段切换处理函数
- 必须包含四个阶段标识：
  - 复杂度识别
  - 模型路由
  - 预算控制
  - 可观测性

## 风险与控制

### 风险 1：把组件做成纯报表，没有决策链

控制：

- 每个阶段都必须围绕“这一步在做什么决定”展开

### 风险 2：现有成本条形图被弱化

控制：

- 把它保留在预算控制阶段作为核心视图

### 风险 3：与正文术语脱节

控制：

- 阶段名称直接使用正文主链词汇
- 说明文案围绕复杂度、路由、预算、观测四个关键词
