# Practice Playground 设计文档

> 日期：2026-03-21
> 范围：实践篇独立在线 Playground 第一版
> 状态：已完成设计确认，待进入实现规划

## 1. 背景

实践篇当前已经具备两条明确路径：

- 本地命令行运行：`bun run pxx-*.ts`
- 文档阅读路径：按章节阅读原理与代码片段

但它还缺一层更低门槛的体验：用户希望在浏览器中直接填写 `API Key / baseURL / model`，不先折腾本地 Bun 环境，也能快速试跑实践示例，判断接口、模型和章节是否可用。

现有代码与文档已经为这件事提供了一部分基础：

- 示例脚本普遍支持 `OPENAI_API_KEY`
- 多个章节已经支持 `OPENAI_BASE_URL` 与 `OPENAI_MODEL`
- 实践篇首页、`setup` 页和 `P1` 已经补入“在线运行模式”文案入口

因此，这次设计的重点不是补一个全新的能力体系，而是把已有的“运行参数能力”提升为一套独立、清晰、可复用的浏览器实验台。

## 2. 目标

第一版 Playground 要解决的是：

1. 用户可以在浏览器中直接填写 `API Key / baseURL / model`
2. 配置允许保存在浏览器本地存储，方便重复使用
3. 用户无需编辑整段代码，只需选择章节并点击“一键运行”
4. Playground 作为独立页面存在，不打断正文结构
5. 章节页可以直接跳入 Playground，并自动选中当前章节

## 3. 非目标

第一版明确不做以下内容：

- 不做完整在线 TypeScript 编辑器
- 不做服务端代理转发
- 不做所有实践章节一次性接入
- 不做原始 JSON 响应面板
- 不做跨章节保存运行结果
- 不把现有 `practice/*.ts` 命令行脚本直接搬到浏览器执行

换句话说，第一版的定位是“教学型浏览器实验台”，不是“在线 IDE”。

## 4. 入口与信息架构

### 4.1 页面形态

新增独立页面：

- `/practice/playground/`

Playground 与正文分工如下：

- 章节正文：负责解释原理、讲解代码、保留本地命令行路径
- Playground：负责浏览器侧配置、预置示例运行、结果展示

### 4.2 入口流

每个接入章节都在正文内提供“在线运行”入口按钮。

入口跳转格式统一为：

- `/practice/playground/?chapter=<chapter-id>`

例如：

- `/practice/playground/?chapter=p01-minimal-agent`

Playground 页面加载后：

1. 读取 `chapter` 查询参数
2. 自动高亮左侧对应章节
3. 加载该章节的预置示例
4. 在右侧顶部保留“返回原文章”链接

### 4.3 第一批接入章节

第一版只接入以下 5 个章节：

- `P1` 最小 Agent
- `P2` 多轮对话
- `P3` 流式输出
- `P10` ReAct Loop
- `P18` 多模型路由与成本控制

选择理由：

- 都适合“预置示例 + 一键运行”
- 能覆盖最小调用、多轮、流式、推理链路、模型路由这几类核心体验
- 不需要像 `P14` 那样依赖双进程或额外服务

## 5. 页面布局

### 5.1 布局方案

已确认采用 `C1`：

- 左侧章节列表
- 右侧运行主区

### 5.2 左侧章节列表

左侧用于承载章节导航，适合后续继续扩章节，符合“独立实验台”的心智。

左侧信息包括：

- 章节编号
- 章节标题
- 当前选中状态

### 5.3 右侧运行主区

右侧采用固定 4 模块结构，所有章节共用同一骨架：

1. 章节头部
2. 配置区
3. 运行区
4. 结果区

## 6. 右侧模块设计

### 6.1 章节头部

展示内容：

- 当前章节标题
- 一句话目标
- 难度标签
- `返回原文章` 链接
- `查看实践环境准备` 链接

作用：

- 保持 Playground 与正文语境连接
- 用户不会迷失在独立页中

### 6.2 配置区

固定输入项：

- `API Key`
- `baseURL`
- `model`

交互能力：

- 保存到本地
- 清空配置
- 显示/隐藏 `API Key`

配置区是全局共享配置，不按章节拆分存储。第一版默认认为这 5 个章节会共用同一套模型配置。

### 6.3 运行区

运行区显示：

- 当前章节的预置示例说明
- 当前章节的输入摘要
- `一键运行` 按钮

第一版只支持：

- 预置示例
- 固定运行逻辑

第一版不支持：

- 整段代码编辑
- 原始请求体编辑
- 任意自定义脚本执行

### 6.4 结果区

结果区固定包含三块：

1. 请求配置摘要
2. 运行输出
3. 调试信息

其中：

- 请求配置摘要显示当前 `baseURL` 与 `model`
- 运行输出显示模型最终回复
- 调试信息显示请求状态、耗时、错误信息

对于 `P3` 这样的流式章节，输出区按流式方式逐步追加内容。

## 7. 本地存储策略

### 7.1 存储范围

本地存储只保存全局配置，不保存运行结果。

建议使用统一 key，例如：

- `practice-playground-config`

建议结构：

```ts
{
  apiKey: '...',
  baseURL: 'https://...',
  model: 'gpt-4o',
  updatedAt: 1710000000000
}
```

### 7.2 为什么不保存运行结果

不保存运行结果是刻意收敛：

- 避免把模型输出和错误信息长期堆在浏览器本地
- 降低存储结构复杂度
- 防止 Playground 很快膨胀成“会话历史系统”

### 7.3 安全提示

Playground 内需明确提示：

- `API Key` 仅保存在当前浏览器本地存储
- 不会提交到仓库
- 不建议在公共设备上保存
- 可随时通过“清空配置”删除

## 8. 技术实现边界

### 8.1 不复用 CLI 脚本执行

第一版不直接把 `practice/*.ts` 拿到浏览器里跑。

原因：

- 现有脚本面向 Node/Bun 命令行
- 浏览器与 Node 运行时边界不同
- 强行复用会混入环境变量、Node API 与流式实现差异

因此，Playground 需要单独建立一层“浏览器友好的运行适配器”。

### 8.2 浏览器请求策略

浏览器直接发起兼容 OpenAI 接口的 HTTP 请求：

- 通过 `fetch` 请求 `baseURL`
- 使用 `API Key` 做请求头鉴权
- 使用 `model` 作为当前模型参数

第一版按能力类型拆分 runner，而不是“一章一个大组件”。

### 8.3 runner 类型划分

建议抽象为 5 类：

1. `P1`：单次工具调用演示
2. `P2`：预置多轮对话
3. `P3`：流式输出
4. `P10`：简化版 ReAct 链路
5. `P18`：轻量模型路由演示

### 8.4 章节适配策略

#### P1

- 预置天气问题
- 展示工具链路后的最终回答

#### P2

- 内置一组消息历史
- 展示多轮上下文效果

#### P3

- 使用流式响应
- 输出区逐步追加内容

#### P10

- 不追求完整 CLI 版 ReAct 仿真
- 保留“思考 -> 工具 -> 回复”的教学型简化链路

#### P18

- 提供轻量路由规则
- 重点展示“当前为什么选这个模型”
- 不做完整预算系统或成本平台

## 9. 错误处理边界

第一版明确覆盖以下错误：

- `API Key` 缺失
- `baseURL` 不可达
- `model` 无效
- 流式响应中断

处理原则：

- 缺关键配置时阻止运行
- 错误信息直接显示在调试信息区
- 流式中断时保留已收到内容
- 第一版不做自动重试，避免浏览器端重复请求

## 10. 文件拆分建议

建议拆成以下单元：

### 页面与组件

- `PracticePlaygroundShell.vue`
- `PracticePlaygroundSidebar.vue`
- `PracticePlaygroundConfigPanel.vue`
- `PracticePlaygroundRunner.vue`
- `PracticePlaygroundOutput.vue`

### 配置与运行层

- `practicePlaygroundCatalog.ts`
- `practicePlaygroundStorage.ts`
- `practicePlaygroundRunners.ts`
- `practicePlaygroundTypes.ts`

## 11. 实施顺序

建议按以下顺序落地：

1. 搭建独立页骨架与左侧章节切换
2. 接入配置区与 localStorage
3. 先打通 `P1` runner，验证“浏览器直连请求”主链路
4. 追加 `P2 / P3 / P10 / P18`
5. 回填各章节入口按钮与文案
6. 运行构建验证，并补一轮手工浏览器验证

## 12. 最终结论

第一版 Playground 的目标是：

> 做一个独立的实践篇在线实验台，支持 `P1 / P2 / P3 / P10 / P18` 五个章节，用户在浏览器中填写 `API Key / baseURL / model` 后即可一键运行预置示例；配置保存到本地存储，章节页可直接跳入并自动选中当前章节。

这是一个刻意收敛的第一版：

- 它优先解决“先跑起来”
- 不追求“在线改一切”
- 保留现有命令行正文主线
- 为后续扩更多章节留下清晰的复用边界
