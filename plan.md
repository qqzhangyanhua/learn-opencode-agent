# 项目收口与升级执行计划

## 目标

把当前项目从“内容很多的技术文档站”收口为“定位清晰、可信度稳定、可持续维护的 AI Coding Agent 学习产品”。

## 总原则

- 先收口可信度，再优化体验
- 先统一结构，再补自动化治理
- 先解决高影响问题，再做性能和样式细节
- 每个阶段完成后都执行一次验证，不跨阶段堆积风险

---

## 阶段 1：可信度与发布基线收口

### 任务 1.1 统一源码基线口径
文件：
- `.vitepress/config.mts`
- `README.md`
- `docs/version-notes.md`

目标：
- 把默认源码引用从 `dev` 浮动分支改成 commit 锚定策略
- 建立统一的源码链接常量与引用规则
- 保留 `dev` 仅作为“追踪最新实现”的补充入口

执行步骤：
1. 在 `.vitepress/config.mts` 中增加统一的 `sourceCommit`、`sourceTreeUrl`、`sourceBlobUrl`
2. 将全站默认源码仓库入口从 `tree/dev` 改为 commit 锚定链接
3. 在 `README.md` 中同步修改 OpenCode 源码引用口径
4. 在 `docs/version-notes.md` 中重写“源码基线”与“链接策略”说明
5. 明确规定：首页、阅读地图、核心章节、源码快照卡优先使用 commit 锚定

验收标准：
- 配置文件中不再把 `dev` 当默认可信基线
- README、版本说明、站点配置三者口径一致
- `bun run build` 通过

---

### 任务 1.2 清理未完成痕迹与草稿语气
文件：
- `docs/00-what-is-ai-agent/index.md`
- 其他搜到占位词的文档页

目标：
- 移除发布内容中的草稿痕迹
- 清理“需要补充”“TODO”“待补”等作者内部标记

执行步骤：
1. 全仓搜索占位词：`TODO`、`待补`、`需要补充`、`TBD`、`占位`
2. 优先处理首页链路上的前置章节
3. 把无法立刻补完的内容改写成正式文案，而不是保留作者备注
4. 复查章节标题和段落语气，避免“讲义式草稿感”

验收标准：
- 前置章节不再出现明显未完成标记
- 首章读感达到“可发布内容”标准
- 搜索结果中不再出现高风险占位词

---

### 任务 1.3 升级封版清单为发布 SOP
文件：
- `docs/release-checklist.md`

目标：
- 把当前“状态描述”升级成“可执行发布流程”

执行步骤：
1. 将清单拆为：
   - 发布阻塞项
   - 非阻塞优化项
   - 发布前验证流程
2. 给每项增加验证方式
3. 把“仍需确认”类条目改成可勾选任务
4. 明确哪些未完成项会阻止发版

验收标准：
- `release-checklist` 可直接作为发版流程使用
- 阻塞项与非阻塞项边界清晰
- 新人也能按文档完成一次发布前检查

---

## 阶段 2：重构项目主叙事与学习路径

### 任务 2.1 首页定位收束
文件：
- `docs/index.md`

目标：
- 让用户 10 秒内理解项目的三条主线与目标人群

执行步骤：
1. 在 hero 区或“学习方式”之前增加一句核心定位
2. 明确三条线的关系：
   - 理论篇：看别人怎么实现
   - 实践篇：自己怎么实现
   - 中级篇：工程问题怎么落地
3. 检查 CTA 顺序是否符合学习路径
4. 压缩重复表述，减少首页信息噪音

验收标准：
- 首页从“信息总览”升级为“产品入口”
- 三条主线关系清楚
- 首页文案不再重复分散

---

### 任务 2.2 建立理论篇 / 实践篇 / 中级篇映射
文件：
- `docs/reading-map.md`

目标：
- 打通当前三条内容线，解决“栏目平行但关系不强”的问题

执行步骤：
1. 新增“章节映射总表”
2. 每行至少包含：
   - 理论章节
   - 对应实践项目
   - 对应中级专题
3. 在推荐阅读路线里说明三条线如何汇合
4. 保持映射表简洁，不追求机械一一对应

验收标准：
- 用户能快速看懂三条线如何互相支撑
- 阅读地图兼具“阅读顺序”和“体系导航”功能
- 理论与实践的桥接明显增强

---

### 任务 2.3 强化实践篇入口的执行属性
文件：
- `docs/practice/index.md`
- `practice/README.md`
- `docs/intermediate/index.md`

目标：
- 让实践篇和中级篇的角色更明确，减少用户理解成本

执行步骤：
1. 在 `docs/practice/index.md` 页首补充“实践篇与理论篇的关系”
2. 统一实践命令格式，只保留一种标准写法
3. 对 `P14` 等特殊章节补充分步运行说明
4. 在 `practice/README.md` 中同步站内说明
5. 在 `docs/intermediate/index.md` 中明确中级篇是“理论+实践之后的工程深化”

验收标准：
- 站内实践页与仓库实践 README 口径一致
- 特殊章节的运行边界清晰
- 中级篇不再像额外目录，而是学习闭环的一部分

---

## 阶段 3：建立最小自动化治理

### 任务 3.1 为内容项目补基础检查脚本
文件：
- `package.json`
- `scripts/check-content.mjs`
- `scripts/check-practice-entries.mjs`

目标：
- 建立最小自动化门禁，降低人工维护成本

执行步骤：
1. 在 `package.json` 中增加脚本：
   - `check:content`
   - `check:practice`
   - `typecheck`
   - `build:strict`
2. 新建 `scripts/check-content.mjs`，检查：
   - 占位词
   - 关键页面存在性
   - 可扩展的 frontmatter/结构问题
3. 新建 `scripts/check-practice-entries.mjs`，检查：
   - `docs/practice` 中引用的 `pxx-*.ts` 是否真实存在
   - 特殊章节是否有额外文件说明
4. 在 `release-checklist` 中加入自动化检查要求

验收标准：
- 这些脚本可本地实际运行
- 至少能发现占位词与实践入口不一致问题
- `package.json` 不再只有 dev/build/preview/start

---

### 任务 3.2 收口仓库元信息与术语单一来源
文件：
- `package.json`
- `docs/glossary.md`

目标：
- 提升仓库对外识别度，统一关键术语口径

执行步骤：
1. 将 `package.json` 的 `name` 从 `book` 改为更明确的项目名
2. 补充必要的描述字段
3. 在 `docs/glossary.md` 中统一：
   - Agent
   - Subagent
   - Primary Agent
   - 运行时
   - 工作台
4. 复查首页、阅读地图、版本说明中的高频术语

验收标准：
- 仓库元信息更专业
- 核心术语不再多口径
- 术语表成为统一定义来源

## 阶段 4：主题层与性能收口

### 任务 4.1 控制主题全局注册与首页交互负担
文件：
- `.vitepress/theme/index.ts`
- `.vitepress/theme/components/PracticePreview.vue`
- `.vitepress/theme/components/LearningPath.vue`
- `.vitepress/theme/components/RuntimeLifecycleDiagram.vue`

目标：
- 降低文档站主题层的维护成本和运行时负担

执行步骤：
1. 检查 `.vitepress/theme/index.ts` 中是否全量注册过多组件
2. 能局部引入的组件改为局部引入
3. 优化首页重组件，减少不必要的响应式逻辑
4. 保持首页教学价值，但降低前端负担

验收标准：
- 全局组件注册更克制
- 首页关键组件结构更简洁
- `bun run build` 通过

---

### 任务 4.2 优化动画与静态站点包体风险
文件：
- `.vitepress/theme/components/animations/core/LottiePlayer.vue`
- `.vitepress/theme/custom.css`

目标：
- 对构建警告暴露出的风险做第一轮治理

执行步骤：
1. 为 Lottie 做懒加载、可见时加载或降级占位
2. 检查是否所有动画都值得保留
3. 收口移动端、长代码块、表格溢出、按钮组布局等问题
4. 重点检查首页与长文页的排版稳定性

验收标准：
- 动画组件不再无条件加载
- 首页和关键页面移动端体验更稳
- 构建警告有明确改善方向

---

## 阶段 5：发布候选版验收

### 任务 5.1 完整验证与发布候选收口
文件：
- `docs/release-checklist.md`
- 相关被修改页面

目标：
- 输出一个“可发布候选版”，并明确剩余阻塞项

执行步骤：
1. 运行：
   - `bun run build`
   - `bun run check:content`
   - `bun run check:practice`
2. 人工抽查页面：
   - 首页
   - 阅读地图
   - 实践首页
   - 中级篇首页
   - 2 至 3 个核心章节
3. 根据结果更新 `release-checklist`
4. 标记当前版本的阻塞项与非阻塞项

验收标准：
- 项目达到“候选发布版”状态
- 有清晰的剩余问题清单
- 后续迭代不会再次失控

## 执行顺序

1. 阶段 1：先解决可信度与草稿痕迹
2. 阶段 2：再统一主叙事与路径设计
3. 阶段 3：补最小治理能力
4. 阶段 4：最后处理主题层与性能
5. 阶段 5：统一验收并收口

---

## 每阶段固定验证

### 阶段 1 结束
- `bun run build`

### 阶段 2 结束
- 人工抽查：首页、阅读地图、实践页、中级篇入口

- 执行自动化检查脚本

### 阶段 4 结束
- `bun run build`
- 重点检查首页和移动端页面

### 阶段 5 结束
- 完整发布候选版验收
