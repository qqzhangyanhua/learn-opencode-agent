# 产品化 AI Agent 学习站研究：常见风险与陷阱

**研究日期：** 2026-03-21
**项目阶段：** Subsequent milestone
**研究问题：** 把现有文档仓库升级为产品化学习站时，最常见的错误是什么？应该在哪个阶段预防？

## 结论摘要

这类项目最容易犯的错不是技术做不出来，而是**产品目标和内容结构没有同步升级**。换句话说，如果只做页面美化，而不重构学习路径、章节结构和实践闭环，站点依然会像“更好看的文档仓库”。

MDN 的教育者资源强调三件事：
- clear progression
- clear learning objectives
- regular assessments / practical projects

这三件事恰好也是你项目最该防守的地方。

## Pitfall 1：只改视觉，不改学习结构

**表现：**
- 首页更漂亮了，但用户进入第二层页面后仍不知道下一步
- 导航变多了，但路线还是按作者视角组织

**预警信号：**
- 方案讨论主要围绕 hero、卡片、配色
- 很少讨论“目标用户从哪里开始”“为什么读这一章”“读完去哪”

**预防策略：**
- 先定义学习路径和章节模板，再做首页与视觉收口
- 把“用户 30 秒内知道从哪里开始”作为验收标准之一

**应在阶段处理：**
- Phase 1（信息架构 / 路径）

## Pitfall 2：继续让 `config.mts` 承担全部课程结构

**表现：**
- 所有导航、路线、推荐关系都手写在单个配置文件里
- 每新增一章或一条路线都要手改多处

**预警信号：**
- 需求一旦涉及“按人群显示内容”就需要大量手工判断
- 章节信息无法复用到首页、路径页、搜索页

**预防策略：**
- 尽早建立 frontmatter + data loader 的课程元数据层
- 把页面关系从“硬编码”转为“数据驱动”

**应在阶段处理：**
- Phase 1

## Pitfall 3：章节结构不统一

**表现：**
- 有的章节很适合入门，有的章节过于源码导向
- 用户无法从页面表面判断是否适合现在阅读

**预警信号：**
- 用户需要先读一大段正文才知道这章是不是自己要的
- “前置知识”“学习目标”“练习”经常缺失

**预防策略：**
- 建统一章节模板
- 至少给每章补齐：
  - 学习目标
  - 前置知识
  - 实践关联
  - 下一步推荐

**应在阶段处理：**
- Phase 2（章节结构）

## Pitfall 4：实践篇仍然像脚本列表

**表现：**
- 用户能看到 `practice/*.ts`，但不知道应该怎么跟练
- 文档和代码是“同时存在”，但不是“同一个学习任务”

**预警信号：**
- 实践页仍按文件名组织，而不是按学习任务组织
- 页面没有完成标准或推荐顺序

**预防策略：**
- 把实践篇设计成课程索引
- 每个项目标注目标、前置、运行入口、完成标准、下一步

**应在阶段处理：**
- Phase 3（实践体验）

## Pitfall 5：过早引入重功能

**表现：**
- 在路径和内容结构没整理好之前，就讨论登录、AI 助教、评论区

**预警信号：**
- 用户核心痛点还没解决，路线已经开始分叉到账号系统

**预防策略：**
- 严守 v1 边界
- 只做内容产品化和学习体验升级

**应在阶段处理：**
- 全程持续防守

## Pitfall 6：没有可验证的学习闭环

**表现：**
- 站点可以读，但无法判断“我学完了吗”“我该动手了吗”

**预警信号：**
- 章节结尾没有行动项
- 用户只能自己判断是否进入实践篇

**预防策略：**
- 在章节末尾加入：
  - 本章练习
  - 对应实践
  - 下一步建议
- 至少提供轻量的任务闭环，不一定要评分系统

**应在阶段处理：**
- Phase 2 / Phase 3

## 最值得优先防守的 3 个风险

1. 只做 UI，不做 IA
2. 不建元数据层，导致后续不可维护
3. 实践篇没有课程化闭环
